## Documented friction gates

- flyctl is mandatory, with operating-system-specific installation and possible PATH setup. Stage 3. Evidence state: documented requirement. Sources: S3, S4.
- The CLI launches a browser-based signup or login flow; WSL may require a symbolic-link fix for browser launch. Stage 4. Evidence state: documented requirement. Sources: S3, S5.
- GitHub signup adds an email confirmation and password-setting step for later active verification. Stage 4. Evidence state: documented requirement. Sources: S3.
- The demo guide says a credit-card prompt occurs at some point, while the current free trial may defer it. Trial exhaustion stops apps and blocks new launches or deploys until billing is configured. Stage 5. Evidence state: documented requirement. Sources: S3, S6.
- Fly Launch proposes project-dependent settings, and the HelloFly candidate explicitly requires accepting defaults or entering web customization. Stage 7. Evidence state: documented requirement. Sources: S3, S8.
- When `fly launch` deploys the app, that deployment must complete before status and browser verification. Stage 7. Evidence state: documented requirement. Sources: S2, S3.
- When Fly Launch prompts for a separate `fly deploy`, that conditional deployment must complete before status and browser verification. Stage 8. Evidence state: documented requirement. Sources: S2, S3.
- For non-obvious deployment failures, Fly.io documents `fly version update`, `fly doctor`, `fly config validate`, `fly logs`, and optional debug or SSH diagnostics. Stage 8. Evidence state: documented requirement. Sources: S7.
