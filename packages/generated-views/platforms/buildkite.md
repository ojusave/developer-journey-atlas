**Developer goal:** Create a free personal Buildkite account, run the vendor-provided Starter pipeline, and inspect its completed annotation.

**Meaningful success:** Create a personal Buildkite account and organization, select the Buildkite Examples Starter pipeline, run its first build on the default hosted environment, wait for completion, and inspect the annotation.

## Documented path

1. Open Getting started with Pipelines and follow the free personal-account path. Success signal: GitHub and email signup options are available. Sources: S1.
2. Choose Sign up with GitHub. Success signal: Buildkite requests access to the GitHub account. Sources: S1.
3. Authorize Buildkite to access the GitHub account. Success signal: Authorization returns the developer toward Buildkite onboarding. Sources: S1.
4. If GitHub shows Install Buildkite, choose repository access and complete installation. Success signal: The installation completes and the New Pipeline flow becomes available. Sources: S1.
5. Create the default personal Buildkite environment and open New Pipeline. Success signal: New Pipeline is visible. Sources: S1.
6. Switch the New Pipeline Git scope to Buildkite Examples if needed. Success signal: Buildkite Examples appears as the selected scope. Sources: S1.
7. Select Starter pipeline in the Repository field. Success signal: Starter pipeline is selected. Sources: S1.
8. Review the New Pipeline fields and keep tutorial defaults. Success signal: The generated pipeline name, branch, cluster, and YAML are visible. Sources: S1.
9. Inspect the Starter YAML's three steps and dependencies. Success signal: All three steps and their ordering are visible. Sources: S1.
10. Select Create and run. Success signal: The build page opens and jobs begin. Sources: S1.
11. Dispatch and run the Starter build jobs on available agents. Success signal: The build reaches a completed state. Sources: S1.
12. Open the completed build's Annotations tab. Success signal: The Annotations tab opens. Sources: S1.
13. Verify the annotation content. Success signal: The expected annotation content is rendered. Sources: S1.

## Documented friction gates

- Signup branches between GitHub authorization and email/password plus email verification. Stage 2. Evidence state: documented requirement. Sources: S1.
- GitHub signup requires authorizing Buildkite to access the account. Stage 3. Evidence state: documented requirement. Sources: S1.
- Installing the GitHub App in an organization can require repository-owner or administrator permission. Stage 4. Evidence state: documented requirement. Sources: S1.
- GitHub signup preselects a repository, so the developer must switch to Buildkite Examples to follow the normalized path. Stage 6. Evidence state: documented requirement. Sources: S1.
- Pipeline name must not match an existing pipeline name; generated repository, branch, cluster, and YAML choices affect execution. Stage 8. Evidence state: documented requirement. Sources: S1, S2.
- The developer must wait for hosted agents to run the dependency-ordered build before checking annotations. Stage 11. Evidence state: documented requirement. Sources: S1.
