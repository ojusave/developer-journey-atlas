**Developer goal:** Reach the first documented WorkOS success after selecting WorkOS as the platform.

**Meaningful success:** The developer chooses a WorkOS product and route, configures the staging environment and credentials required by that route, executes its first workflow, and verifies the product-specific result.

**Evidence boundary:** demonstrated-terminal-state. Selecting a product changes the required resources, credentials, external systems, API calls, waits, and terminal verification. AuthKit is visually prominent, but treating it as the universal WorkOS journey would omit eleven peer products.

## Documented path

1. Open the WorkOS documentation landing page and inspect the product-level routes. Success signal: The product inventory and separate outcomes are visible. Sources: S1.
2. Choose the WorkOS product and its documented starting route. Success signal: One product, implementation route, and terminal outcome are selected. Sources: S1, S4, S5.
3. Open an existing project supported by the AuthKit installer and verify Node.js 20 or later. Success signal: The shell is in the project and Node.js 20+ is available. Sources: S4.
4. Run npx workos@latest install from the project root. Success signal: The CLI begins framework detection and opens or requests WorkOS authentication. Sources: S4.
5. Complete the browser handoff to create or sign in to the WorkOS account and authorize the CLI for the intended staging project. Success signal: The browser returns control to the CLI and it can read the WorkOS project configuration. Sources: S2, S3, S4.
6. Allow the installer to detect the project's framework and integration shape. Success signal: The CLI identifies a supported framework and presents the planned setup. Sources: S4.
7. Review and confirm the installer prompts for redirect URI, CORS, and homepage URL for the local application. Success signal: The chosen URLs match the application's local runtime. Sources: S4.
8. Allow the installer to install the matching WorkOS SDK. Success signal: The dependency is added to the project's package or dependency manifest. Sources: S4.
9. Allow the installer to analyze the project and create or update the callback routes, authentication middleware or provider wrappers, and environment configuration appropriate to the framework. Success signal: The CLI reports the expected files and configuration changes. Sources: S4.
10. Let the installer run its build validation. Success signal: The project passes the installer's build validation. Sources: S4.
11. Confirm the CLI displays AuthKit is ready and Run npm run dev to get started, then inspect git diff before accepting the generated integration. Success signal: The AuthKit is ready message and npm run dev instruction appear, and the developer understands the files, routes, middleware, and environment changes. Sources: S4.

## Documented friction gates

- Twelve top-level products define different first-success outcomes, and AuthKit adds automatic versus manual and framework choices. Stage 2. Evidence state: documented requirement. Sources: S1, S4, S5.
- A WorkOS account belongs to exactly one team and is required to access the staging project, Dashboard, and personalized credentials. Stage 2. Evidence state: documented requirement. Sources: S2, S5.
- Dashboard role controls whether the developer can manage keys and environment resources; Sandbox Developer is intentionally staging-only. Stage 2. Evidence state: documented requirement. Sources: S2.
- The staging API key is environment-scoped and powerful; AuthKit also needs a Client ID, while the manual session path adds cookie and CSRF secrets. Stage 2. Evidence state: documented requirement. Sources: S5, S6.
- Auth and SSO routes require exact redirect, sign-in, sign-out, CORS, and homepage values; missing sign-out redirect is a documented logout error. Stage 2. Evidence state: documented requirement. Sources: S4, S5.
- Every SDK path requires the selected framework runtime, package manager, and code changes; the AI Installer additionally analyzes and writes the existing project. Stage 2. Evidence state: documented requirement. Sources: S4, S5.
- Staging is free, but production remains locked until billing information is added and is excluded from first staging success. Stage 2. Evidence state: documented requirement. Sources: S3.
