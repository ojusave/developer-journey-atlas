**Developer goal:** Register for SingleStore Helios, create the no-credit Shared workspace, and run a first browser SQL query against its attached database.

**Meaningful success:** A query runs against the database attached to a Free Shared workspace and its result is displayed in SQL Editor.

**Evidence boundary:** demonstrated-terminal-state. A displayed SQL result is the first proof that the free workspace, attached database, browser connection, and query engine are working.

## Documented path

1. Open Getting Started with SingleStore Helios and follow its Cloud Portal registration link for Free Shared Edition. Success signal: The official free entry point is selected. Sources: S1, S2, S7.
2. Sign up at the official SingleStore Cloud Portal. Success signal: Cloud Portal opens for the new account. Sources: S2, S7.
3. Create a Free Shared workspace from Cloud Portal. Success signal: The Shared workspace is available in Cloud Portal. Sources: S1, S3, S4, S10.
4. Open Editor > Open SQL Editor, or select the Shared workspace > Connect > SQL Editor. Success signal: The workspace and database are selectable in the editor. Sources: S1, S5.
5. Create a new SQL file, select the Shared workspace and attached database, enter a documented example query from the Getting Started notebook, select the query, and click Run. Success signal: SQL Editor completes without an error. Sources: S5, S11.
6. Verify that SQL Editor displays the query result set. Success signal: Rows or scalar output appear in the results pane. Sources: S5, S11.

## Documented friction gates

- The static unauthenticated Cloud Portal response contains only an empty client application root. The official signup guide shows the initial direct-email form but does not document its post-Continue journey or social-provider convergence. Stage 2. Evidence state: documented requirement. Sources: S2, S7.
- Shared consumes no credits but disables CREATE DATABASE and is not for production; Standard consumes trial credits but supports the detailed sample-data tutorial's database creation. Stage 3. Evidence state: documented requirement. Sources: S2, S4, S9.
- The workspace guide gives only Create New > Workspace and `Follow the provided instructions`. Shared docs separately fix AWS US-EAST-1, disable memory and vCPU sizing, and require unique database and database-user names, but the exact wizard, waits, and terminal status remain undocumented. Stage 3. Evidence state: documented requirement. Sources: S3, S4.
- The organization guide says every organization must have a billing account to create workspaces and databases, while the Shared and signup guides do not document whether registration supplies one automatically or requires a payment method. Stage 3. Evidence state: documented requirement. Sources: S2, S6.
- The SQL Editor guide documents file navigation and settings, not a first query execution and result for the automatically assigned Shared database. Stage 3. Evidence state: documented requirement. Sources: S5.
- The generic SQL Editor guide says to verify the workspace-group firewall after a connection error, while Shared Edition says firewall access cannot be configured and the database is accessible from anywhere by default. That recovery is not applicable as written to the selected Shared route. Stage 3. Evidence state: documented requirement. Sources: S4, S5.
- Standard trial workspace groups receive an inactivity warning after two query-free days, suspend 24 hours later without a query, and can be resumed through Cloud Portal or Management API. Shared workspaces do not use this inactivity policy. Stage 3. Evidence state: documented requirement. Sources: S2.
