**Developer goal:** Start without assuming a Celonis team, role, license, data source, application, or existing content, identify the official general first-mile route, and follow it to the platform's first documented success.

**Meaningful success:** A new Studio package is created from an existing Data Model with one View and one Knowledge Model.

**Evidence boundary:** demonstrated-terminal-state. The new policy supplies the selection rule: choose the first developer path rather than an administrator-only path. Package creation is the earliest explicit Studio artifact terminal; adding components, testing, publishing, and sharing follow later.

## Documented path

1. Open Getting started with the Celonis Platform and select Creating applications and analyzing data. Success signal: Creating applications and analyzing data links to Studio. Sources: S1.
2. Join a Celonis Platform team through an administrator invitation or an administrator-enabled open-signup URL, then sign in with the team's configured identity method. Success signal: Studio is visible in the team. Sources: S2, S3, S4.
3. Open Studio and choose a space where a package can be created. Success signal: Create package is available. Sources: S6.
4. Click Create package and enter the package settings. Success signal: Create is enabled. Sources: S6.
5. Click Create. Success signal: The package is displayed and the View layout editing screen opens. Sources: S6, S7.

## Documented friction gates

- The general documentation requires a role and task-family choice before a product route can be followed. Stage 2. Evidence state: documented requirement. Sources: S1.
- Every user must belong to at least one Celonis Platform team. Stage 2. Evidence state: documented requirement. Sources: S1.
- Team administrators control user access and permissions, and task capabilities vary by role. Stage 2. Evidence state: documented requirement. Sources: S1, S3.
- Default Celonis ID sign-in requires an access token sent to the user's email address. Stage 2. Evidence state: documented requirement. Sources: S2.
- A team's sign-in method can be Celonis ID, SAML, or OIDC. Stage 2. Evidence state: documented requirement. Sources: S2.
- Product features depend on the team's license, and open signup can consume licensed seats. Stage 2. Evidence state: documented requirement. Sources: S1, S3.
- The current release notes say the prior data-integration Quickstarts were removed, so those old platform options cannot define a current general route. Stage 2. Evidence state: documented requirement. Sources: S5.
