**Developer goal:** Start without a Zoho account, create access for a Zoho product, register a Self Client in the Zoho API Console, choose the product scopes and grant flow, and obtain the first access token for the developer's own account.

**Meaningful success:** A successful Self Client token exchange returns an access token, token type, API domain, and expiration; the authorization-code branch also returns a refresh token.

**Evidence boundary:** explicitly-named. The shared OAuth guide places token acquisition immediately after Self Client creation and before service API calls. Signup is opaque under policy; the developer supplies the service-specific scope required for their own Zoho product.

## Documented path

1. Open the Self Client overview and select the client-credentials flow. Success signal: Create a self client and Get access token are shown. Sources: S1.
2. Create a free Zoho account using the official signup URL and activate the Zoho service whose API scope will be requested. Success signal: The Zoho API Console can be opened. Sources: S4, S5.
3. Open the Zoho API Console, click GET STARTED, hover over Self Client, and click CREATE NOW. Success signal: The CREATE action is available. Sources: S1, S2, S5.
4. Enter the required Client Name, click CREATE, then click OK. Success signal: Both values appear under the Client Secret tab. Sources: S1, S2.
5. Choose the documented OAuth scope for the intended Zoho service and identify the account's data-center-specific Accounts server. Success signal: client_id, client_secret, scope, grant_type, endpoint, and any required soid are available. Sources: S3, S5, S6.
6. POST the client-credentials token request to `{accounts-server-url}/oauth/v2/token`. Success signal: The response includes `access_token`, `api_domain`, `token_type`, and `expires_in`. Sources: S6.
7. Verify the returned access-token fields. Success signal: `access_token`, `api_domain`, `token_type`, and `expires_in` are present. Sources: S1, S6.

## Documented friction gates

- A Zoho account is required, but the public signup response does not expose the complete form or completion sequence. Stage 2. Evidence state: documented requirement. Sources: S4.
- The API Console is behind Zoho Accounts sign-in and can involve external identity, password, OTP, MFA, linked-account, and data-center states. Stage 3. Evidence state: documented requirement. Sources: S5.
- The developer must choose a Zoho service and scope, and Self Client offers two peer token grant flows. Stage 3. Evidence state: documented requirement. Sources: S1, S3, S6.
- Self Client creation produces a Client ID and Client Secret that must be protected and used in the token flow. Stage 3. Evidence state: documented requirement. Sources: S1, S2.
- The authorization-code flow can require selecting the Zoho app and portal, plus a data-center-specific accounts server. Stage 3. Evidence state: documented requirement. Sources: S3, S5.
