**Developer goal:** Create a Bird account and workspace, then use the official cURL quickstart until the sandbox message reaches delivered status.

**Meaningful success:** A deterministic sandbox email sent through Bird's shared onboarding domain is accepted and then reports delivered status without reaching a real mailbox.

**Evidence boundary:** demonstrated-terminal-state. The vendor names this its first-email quickstart and demonstrates delivered as its final verification state before production domain, real-recipient, webhook, and SDK work.

## Documented path

1. Open Bird documentation and select the Email first-send quickstart. Success signal: The page identifies an API-key-to-delivered-email path. Sources: S1, S6.
2. Open Bird signup and select Use password instead for the documented work-email-and-password route. Success signal: The route matches Bird's account-creation guide. Sources: S2, S3.
3. Enter the work email and password and submit signup. Success signal: The confirmation screen requests email verification. Sources: S2, S3.
4. Open Bird's verification email and enter its code, or follow its confirmation link. Success signal: The Bird dashboard opens for the first time. Sources: S2.
5. Land in the automatically available workspace. Success signal: The workspace dashboard is visible without DNS or domain verification. Sources: S2, S8.
6. Open Developers, API keys, choose to create a key, name it, and grant `emails:write`. Success signal: The full key beginning `bk_us1_` or `bk_eu1_` is displayed once. Sources: S4, S5, S6.
7. Copy the full key immediately and export it as `BIRD_API_KEY`. Success signal: The key prefix identifies the matching regional API host. Sources: S4, S5, S6.
8. POST the published email payload to the host matching the key's region. Success signal: The request is durably accepted for asynchronous processing. Sources: S5, S6.
9. Process the accepted sandbox email asynchronously through the delivery-status pipeline. Success signal: The deterministic delivered sandbox address selects the happy-path outcome. Sources: S6, S7.
10. GET the message by its returned `em_` ID until its status is delivered. Success signal: The deterministic first-email terminal is observable without a real mailbox. Sources: S6, S7.

## Documented friction gates

- Submitting signup signifies agreement to Bird's Terms and Privacy Policy. Stage 3. Evidence state: documented requirement. Sources: S3.
- Email confirmation is required before the account has normal dashboard use. Stage 4. Evidence state: documented requirement. Sources: S2.
- Key creation requires an owner, Admin, or Developer role, and the first-send key needs `emails:write`. Stage 6. Evidence state: documented requirement. Sources: S4, S5, S8.
- The full API key is displayed only once and must be copied immediately. Stage 7. Evidence state: documented requirement. Sources: S4, S5, S6.
- The API host must match the region encoded in the key prefix or the request returns 421. Stage 8. Evidence state: documented requirement. Sources: S5, S6.
- HTTP 202 means accepted, not delivered; the status must progress asynchronously before verification. Stage 9. Evidence state: documented requirement. Sources: S6, S7.
- The shared onboarding sender is capped at roughly 50 recipients per organization per day and returns 429 above the cap. Stage 9. Evidence state: documented requirement. Sources: S6.
