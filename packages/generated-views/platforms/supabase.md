## Documented path

1. Open the Supabase Getting Started page. Success signal: The documented route groups and choices are visible. Sources: S1.
2. Choose the route whose documented outcome should define first success. Success signal: A human has selected one official route. Sources: S1.
3. Sign in to the Supabase Dashboard and create a new Supabase project from an organization the developer belongs to. Success signal: The project is created and begins launching its database. Sources: S2, S6.
4. Wait until the new Supabase project and database are up and running. Success signal: The project Dashboard and SQL Editor are available. Sources: S2.
5. Open the project's SQL Editor, paste the supplied instruments SQL, and select Run. Success signal: The SQL Editor reports successful execution and the table contains the three sample rows. Sources: S2.
6. If Data API was disabled during project setup, open Integrations > Data API, enable it, and expose the instruments table. Success signal: The instruments table is exposed to the Data API. Sources: S2.
7. Run npm create vite@latest my-app -- --template react. Success signal: The my-app project is created. Sources: S2.
8. Run cd my-app && npm install @supabase/supabase-js. Success signal: @supabase/supabase-js is installed in my-app. Sources: S2.
9. Open the project's Connect panel and copy the Project URL and publishable key. Success signal: Both values are available for the Vite environment file. Sources: S2, S5.
10. Create .env.local in my-app and set VITE_SUPABASE_URL to the Project URL and VITE_SUPABASE_PUBLISHABLE_KEY to the publishable key. Success signal: Both documented Vite variables are present with the project's values. Sources: S2.
11. Replace src/App.jsx with the quickstart code that creates the Supabase client, selects all rows from instruments in getInstruments, stores the returned data, handles errors, and renders each instrument name in a list. Success signal: App.jsx imports createClient, calls supabase.from('instruments').select(), and maps the result to list items. Sources: S2.
12. Run npm run dev from my-app. Success signal: Vite serves the app at http://localhost:5173. Sources: S2.
13. Open http://localhost:5173 and verify the instrument list. Success signal: The browser displays violin, viola, and cello from the Supabase database. Sources: S2.

## Documented friction gates

- The general Getting Started page requires a route, framework, and application-outcome choice that changes the entire first-mile path. Stage 2. Evidence state: documented requirement. Sources: S1, S2, S3, S4, S5.
- Hosted quickstarts require a Supabase project in an organization and a provisioning wait for the new database to launch; the local CLI route does not require that same hosted-project starting state. Stage 2. Evidence state: documented requirement. Sources: S2, S3.
- The Free Plan allows two active projects, with the organization quota affected by every Owner or Admin member. Each organization has one plan, so free and paid projects cannot be mixed in the same organization. Stage 2. Evidence state: documented requirement. Sources: S6, S8.
- A hosted quickstart waits for database provisioning. Separately, a low-activity Free Plan project can receive a warning email and be paused after the documented seven-day activity window. Stage 2. Evidence state: documented requirement. Sources: S2, S7, S8.
- Local and framework routes add different CLI, container, language, framework, and package installation requirements. Stage 2. Evidence state: documented requirement. Sources: S2, S3.
- Hosted client applications need the project URL and an appropriate API key; public and secret keys have materially different placement and security rules. Stage 2. Evidence state: documented requirement. Sources: S2, S5.
