**Developer goal:** Create an Akamai Cloud account, provision a first Linux virtual machine, and establish an interactive remote shell on it.

**Meaningful success:** Akamai Cloud has provisioned and fully booted the first Linode, and the developer has connected to its Linux shell through SSH or the built-in Lish console.

**Evidence boundary:** explicitly-named. Get started with Linodes places Connect to the Linode immediately after creation and begins Deploy new applications with the statement that the Linode can now be used. Application installation, OS hardening, DNS, and workload deployment are therefore after this first usable infrastructure success.

## Documented friction gates

- Email confirmation is mandatory before identity verification can finish. Stage 3. Evidence state: documented requirement. Sources: S1.
- SMS phone verification is a mandatory fraud-prevention gate. Stage 4. Evidence state: documented requirement. Sources: S1.
- A payment method and billing address are required during signup even before a paid service is deployed. Stage 6. Evidence state: documented requirement. Sources: S1.
- A small number of accounts require an undocumented-duration manual activation review. Stage 7. Evidence state: documented requirement. Sources: S1.
- Recommended passwordless access adds local key generation, private-key retention, and a passphrase decision. Stage 8. Evidence state: documented requirement. Sources: S4, S5.
- The developer must choose among multiple plan families and sizes whose availability and pricing vary by region. Stage 13. Evidence state: documented requirement. Sources: S2, S7.
- A safe first public instance requires firewall judgment. Skipping the firewall is possible but explicitly not recommended; an overly restrictive firewall can block SSH. Stage 17. Evidence state: documented requirement. Sources: S2, S6.
- A Distribution image takes about 3 minutes to provision before connection. Stage 19. Evidence state: documented requirement. Sources: S2.
- Lish is the documented out-of-band fallback for network or SSH failure and can be used to inspect boot, sshd, firewall, root-login, and authorized-key state. Stage 20. Evidence state: documented requirement. Sources: S6.

## Unknowns

- What additional evidence, duration, and decision criteria apply when a new account is routed to manual review? Why unresolved: The current Get started guide acknowledges manual review but does not document its steps or timing. Next evidence needed: A minority of users encounter an unbounded gate before the first service can be created.
- Which Distribution and exact plan should be considered the universal tutorial default? Why unresolved: The general creation guide deliberately allows any supported Distribution and suitable regional plan; it provides examples but does not designate one canonical pair. Next evidence needed: The selected OS packages, hourly price, and instance capacity vary between valid first paths.
- Does every signup entry point currently receive the advertised US$100 promotional credit, and for how long? Why unresolved: The free-offer page makes the offer subject to separate promotion rules, while the universal account guide does not state that a credit is automatic. Next evidence needed: The effective cost before first success and urgency of cleanup can differ.
