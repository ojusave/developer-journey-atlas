**Developer goal:** Create, evaluate, and visibly toggle a first LaunchDarkly feature flag without a preselected SDK or application stack.

**Meaningful success:** In a selected SDK branch, an application initializes LaunchDarkly, evaluates a first flag for a context, and changes its visible behavior when the flag is turned on and off.

**Evidence boundary:** explicitly-named. The success outcome is documented, but an atomic path cannot be fixed until a human selects the SDK and application environment.

## Documented path

1. Sign up for the 14-day LaunchDarkly free trial and open the initial project and Test environment. Success signal: The Flags list is accessible for the new account. Sources: S5, S6.
2. Choose the language or framework used by the runnable sample and install its LaunchDarkly SDK with the corresponding package manager. Success signal: The SDK package imports or loads in the selected runtime. Sources: S5, S4.
3. Create the first Boolean feature flag in LaunchDarkly and copy its flag key. Success signal: The flag appears in the project's Flags list and its key can be copied. Sources: S5.
4. Copy the Test environment credential appropriate to the SDK: server-side SDK key, client-side ID, or mobile key. Success signal: The credential is available to the sample application. Sources: S4, S5.
5. Initialize the selected LaunchDarkly client with the Test credential and define the documented user context. Success signal: The application reaches the SDK initialized callback or equivalent ready state. Sources: S2, S5.
6. Run the application and confirm the initialization output. Success signal: The output contains `SDK successfully initialized!` rather than an initialization failure. Sources: S3, S5.
7. Evaluate the first flag for the context with the copied flag key and a `false` fallback, then run the code. Success signal: The application prints the documented true result. Sources: S5.
8. In the Test environment, toggle the flag Off, select Review and save, enter a comment explaining the change, and select Save changes. Success signal: The flag list shows Off for the Test environment. Sources: S3, S5.
9. Run the evaluation code again and verify that the saved flag change changes the returned value. Success signal: The application prints `Our first feature flag is: false` after previously printing true. Sources: S3, S5.

## Documented friction gates

- Self-service signup starts a 14-day no-card trial that automatically converts to the free Developer plan unless upgraded. Stage 1. Evidence state: documented requirement. Sources: S5, S6.
- SDK type, runtime, and application must be selected before copying the corresponding implementation. Stage 2. Evidence state: documented requirement. Sources: S1, S2, S4.
- The application must install the selected LaunchDarkly SDK through its dependency manager. Stage 2. Evidence state: documented requirement. Sources: S2.
- Server-side SDK key, client-side ID, or mobile key must match the SDK and environment. Stage 4. Evidence state: documented requirement. Sources: S2, S4.
- Flag key, context, fallback value, and toggle environment must be consistent between application and UI. Stage 7. Evidence state: documented requirement. Sources: S2, S3.
- Streaming connections, HTTPS proxies, and outbound network authorization can delay or block updates. Stage 9. Evidence state: documented requirement. Sources: S2, S3.
- The application must show the corresponding flag variation after Review and save. Stage 9. Evidence state: documented requirement. Sources: S3.
