**Developer goal:** Create a Pipedrive trial and company, retrieve the company-specific API token, and make the official direct cURL request to list deals.

**Meaningful success:** An authenticated `GET https://{companydomain}.pipedrive.com/api/v2/deals` request returns HTTP 200 and a JSON body whose `success` field is true, whether the data array contains deals or is empty.

**Evidence boundary:** explicitly-named. This is the earliest observable terminal in the selected official quickstart. The workshop policy permits treating signup as one opaque step and then reconstructing the publicly documented post-login path.

## Documented path

1. Open Pipedrive’s official API-token authentication guide. Success signal: The guide shows the `x-api-token` header and endpoint. Sources: S4.
2. Create the no-credit-card Pipedrive trial and company as one opaque signup step. Success signal: The Pipedrive web app opens. Sources: S2.
3. Open account name > Company settings > Personal preferences > API and copy the user’s API token. Success signal: The API tab displays the token. Sources: S3, S4.
4. Run the documented cURL `GET https://{companydomain}.pipedrive.com/api/v2/deals` with the token in `x-api-token`. Success signal: The HTTP response is 200. Sources: S4, S5.
5. Inspect the JSON response envelope. Success signal: The response has `success: true`; `data` may be populated or empty. Sources: S5.

## Documented friction gates

- Email signup requires a work email and the public page is protected by reCAPTCHA. Stage 2. Evidence state: documented requirement. Sources: S2.
- Signing up accepts Pipedrive's Terms of Service and Privacy Notice. Stage 2. Evidence state: documented requirement. Sources: S2.
- Work email and Google are peer initial signup methods with different identity paths. Stage 2. Evidence state: documented requirement. Sources: S2.
- Company creation, authenticated onboarding, company domain, and settings convergence are not exposed by the public registration response. Stage 3. Evidence state: documented requirement. Sources: S2, S3.
- API may be disabled for a company user, in which case the API tab is absent and must be enabled before token retrieval. Stage 3. Evidence state: documented requirement. Sources: S3.
- The API token is company-specific, exposes that user's data, and only one token can be active at a time. Stage 3. Evidence state: documented requirement. Sources: S3, S4.
- The selected first-request route requires Postman plus a Pipedrive API collection and inherited header authentication. Stage 3. Evidence state: documented requirement. Sources: S1.
