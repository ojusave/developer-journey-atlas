**Developer goal:** Create a Cloudinary account, find product-environment credentials, complete the Image and Video Developer Kickstart, and run one official SDK quick start through its demonstrated media-delivery terminal.

**Meaningful success:** Run a selected official SDK quick start end to end and observe its documented media output.

**Evidence boundary:** explicitly-named. The Developer Kickstart uses successful execution of a selected SDK quick start as its final required progress marker. The Node.js candidate demonstrates the corresponding end-to-end output as a transformed image loaded from the generated URL.

## Documented path

1. Open the Developer Kickstart and begin its ordered onboarding tasks. Success signal: The page displays the progress control 'Got it - I'm ready to get started'. Sources: S1.
2. Register for a free Cloudinary account using Google, GitHub, or email. Success signal: For email registration, Cloudinary sends a verification email; the account becomes usable only after its link is opened. Sources: S2, S3, S4.
3. Open Console Settings, navigate to API Keys, and locate the product environment cloud name, API key, API secret, and API environment variable. Success signal: The Kickstart progress markers for having an account and knowing where to find credentials can be checked. Sources: S2, S5.
4. Complete the Kickstart's ordered optimization, first-transformation, and resize tasks, then continue to Step 5. Success signal: The Step 5 page presents backend, frontend, and mobile quick-start groups. Sources: S8, S9, S10, S11.
5. Choose Sign Up with Email, enter full name, email, and a compliant password, then select Get Started Now. Success signal: The official help article says the verification email is sent automatically. Sources: S3, S4.
6. Open the verification link from the Cloudinary email to complete signup and activate the account. Success signal: The account can use Cloudinary APIs and tools after verification. Sources: S4.
7. Open Console Settings > API Keys and copy the API environment-variable format, replacing its API-key and API-secret placeholders with the displayed values. Success signal: Cloud name, API key, API secret, and environment-variable format are visible. Sources: S2, S5, S6.
8. Open the Step 2 sample image delivery URL with the account cloud name and inspect the original image. Success signal: The high-resolution JPG is visible in the browser. Sources: S8.
9. Insert `/q_auto/f_auto` after `/upload`, reload the image, inspect its size and format, and compare it with the original. Success signal: The guide says inspection shows roughly one third or less of the original file size without an obvious visual-quality difference. Sources: S8.
10. Insert `/c_scale,w_500` before `/q_auto/f_auto`, reload the image, and inspect the delivered size and format. Success signal: The resized delivery is visible and its file size and format can be inspected. Sources: S8.
11. Open the sample video with the account cloud name, inspect the original, add the same scale and automatic optimization parameters, reload it, then let it play to the end before checking the result. Success signal: The developer can compare the optimized video's format and file size with the original. Sources: S8.
12. Add `/e_grayscale` before `/c_scale,w_500` and reload the Step 3 image URL. Success signal: The effect transformation is visible. Sources: S9.
13. Add `/bo_40px_solid_green` after `/e_grayscale` and reload the URL. Success signal: The border transformation is visible. Sources: S9.
14. Add the published Good morning text-layer and layer-apply components after the border, remove `e_grayscale`, and reload the URL. Success signal: The border and Good morning text are visible on the color image. Sources: S9.
15. Remove `/f_auto`, change the extension from `.jpg` to `.png`, reload and inspect the PNG, then restore `/f_auto` and reload again. Success signal: Inspection first shows PNG; after restoring `f_auto`, Cloudinary selects a browser-appropriate format. Sources: S9.
16. Replace `c_scale,w_500` with each documented automatic crop or pad transformation, then keep the predominant-gradient variant and test it against each listed sample public ID. Success signal: The developer sees the impact of the different automatic resize transformations. Sources: S10.
17. Return to the simple scale-cropping URL, replace its resize component with each documented face-detection crop, and inspect both results. Success signal: The developer sees how face-detection gravity preserves people during resizing. Sources: S10.
18. In a supported Node.js environment, run `npm install cloudinary`. Success signal: npm completes the package installation. Sources: S6.
19. Set the CLOUDINARY_URL environment variable for the current platform using the copied credential values. Success signal: The later script logs the expected cloud_name, api_key, api_secret, and secure configuration. Sources: S6.
20. Create index.js, require Cloudinary v2, enable secure URLs, and add the published uploadImage, getAssetInfo, and createImageTag functions. Success signal: The main function uploads the sample, retrieves its colors, creates an image tag, and logs it. Sources: S6.
21. Save index.js and run `node index.js`. Success signal: The upload response includes public_id and secure_url, the resource response includes colors, and the generated tag contains the transformation URL. Sources: S6.
22. Open the transformed URL from the generated image tag in a browser. Success signal: The browser displays the circular face thumbnail with the documented outline and background transformations. Sources: S6.

## Documented friction gates

- A Cloudinary account is required for the Kickstart and SDK route. Stage 2. Evidence state: documented requirement. Sources: S2, S3.
- Email registration requires clicking a verification link before APIs and tools can be used. Stage 2. Evidence state: documented requirement. Sources: S4.
- The backend SDK route requires a product-environment cloud name, API key, and API secret, with the secret protected from exposure. Stage 3. Evidence state: documented requirement. Sources: S5, S6.
- The official final Kickstart task requires a human choice among materially different backend, frontend, and mobile SDK paths. Stage 5. Evidence state: documented requirement. Sources: S11, S7.
