**Developer goal:** Create an incident.io account through Slack, install the incident.io app, declare a first live incident, and observe the dedicated incident channel and main-channel announcement.

**Meaningful success:** A first live incident has been declared and incident.io has created its dedicated Slack incident channel and posted the default heads-up in the main incidents channel.

**Evidence boundary:** demonstrated-terminal-state. This is the first observable product outcome after identity, installation, and declaration. Video calls, linked tickets, workflow actions, incident coordination, resolution, and post-incident work are conditional or occur after this point.

## Documented path

1. Open the official incident.io help center and locate the getting-started and incident-declaration paths. Success signal: The official Slack installation and declaring-incidents guides can be opened. Sources: S1.
2. Choose Slack or Microsoft Teams as the connected chat surface. Success signal: The chosen chat platform's getting-started guide is open. Sources: S1, S2.
3. Choose the incident mode and declaration interface for the first success. Success signal: The candidate selects a live incident declared by Slack slash command. Sources: S3, S4.
4. Open Installing incident.io in 30 seconds and follow its linked Sign Up entry. Success signal: Sign in with Slack is displayed. Sources: S2.
5. Select Sign in with Slack and choose the intended Slack Workspace, not the Enterprise organization container. Success signal: The intended workspace name is visible on the authorization screen. Sources: S2.
6. Review and authorize the Slack identity permissions requested by incident.io. Success signal: The setup flow advances to integration selection. Sources: S2.
7. Select the integrations the organization uses and continue. Success signal: The Slack installation control is available. Sources: S2.
8. Select Add incident.io to Slack. Success signal: The intended workspace and requested app permissions are shown. Sources: S2.
9. As a Slack administrator, authorize the incident.io app for the intended workspace. Success signal: The install completes and the workspace contains the incident.io app and main incidents channel. Sources: S2.
10. Verify that incident.io is installed and that the main incidents channel exists in Slack. Success signal: The incident.io app is installed and the main incidents channel is visible. Sources: S2.
11. In any non-incident Slack channel, type `/inc Website is down` and press Enter. Success signal: The incident details form is visible. Sources: S3.
12. Keep the incident in live mode and complete every field the form marks as required. Success signal: All required fields are satisfied and Declare is available. Sources: S3.
13. Select Declare. Success signal: A dedicated incident channel is created and a heads-up appears in the main incidents channel. Sources: S3.
14. Open the dedicated incident channel and confirm the main-channel heads-up. Success signal: The dedicated channel is available for responders and the main incidents channel contains the incident announcement. Sources: S3.

## Documented friction gates

- Slack and Microsoft Teams are peer getting-started surfaces with different installation and interaction paths. Stage 2. Evidence state: documented requirement. Sources: S1.
- Slack signup is restricted to corporate or company email addresses. Stage 2. Evidence state: documented requirement. Sources: S2.
- A Slack administrator must authorize the app because of Slack's permission model. Stage 2. Evidence state: documented requirement. Sources: S2.
- Restricted public-channel creation requires a workspace policy change or a connected Slack admin user before incident.io can create required channels. Stage 2. Evidence state: documented requirement. Sources: S2.
- Selecting an Enterprise organization rather than a Workspace can produce the documented `no_bot_scopes_requested` authorization error. Stage 2. Evidence state: documented requirement. Sources: S2.
- Live, test, tutorial, and retrospective modes intentionally change visibility, reporting, and completion signals. Stage 3. Evidence state: documented requirement. Sources: S3, S4.
- Incident forms and required fields are customizable by organization, and admins can restrict access to incident types. Stage 3. Evidence state: documented requirement. Sources: S3.
- Duplicate `/inc` or `/incident` slash commands can invoke the most recently installed Slack app instead of incident.io. Stage 3. Evidence state: documented requirement. Sources: S2.

## Unknowns

- Should the representative incident.io first mile use Slack or Microsoft Teams? Why unresolved: The official help center presents both as peer getting-started routes and does not designate one universal default. Next evidence needed: The choice changes identity, admin authorization, installation, interaction, and terminal communication surfaces.
- Should first success be a live, test, tutorial, or retrospective incident? Why unresolved: The incident-mode documentation explains each mode but leaves the safety and visibility choice to the organization or user. Next evidence needed: The modes intentionally differ in announcements, incident-library visibility, Insights inclusion, and response side effects.
- Which declaration interface should represent the general path? Why unresolved: The declaration guide documents these as alternatives without selecting one for all users. Next evidence needed: Slash command, Slack message, and browser entry require different interactions and inputs before the same or similar incident outcome.
- Which fields and incident types will a fresh organization's form expose? Why unresolved: Incident forms are fully customizable and administrators can restrict incident-type declaration permissions. Next evidence needed: The number of actions and permission checks before Declare can vary.
