**Developer goal:** Use Salesforce's REST API Quick Start to authenticate a free Developer Edition org, inspect REST resources and Account metadata, query Accounts, update one Account's BillingCity, and verify the update.

**Meaningful success:** The developer authenticates a free Salesforce Developer Edition org with Salesforce CLI, walks through the documented REST discovery and Account queries, patches one Account's BillingCity to Fremont, receives HTTP 204, and sees the changed value in Salesforce.

**Evidence boundary:** demonstrated-terminal-state. The Quick Start introduction defines its outcome as updating a record. The final PATCH plus visible Account refresh proves signup, permission, CLI authentication, token retrieval, REST discovery, query, request body, write permission, and the update itself.

## Documented path

1. Open Salesforce's Developer Edition signup page from the REST Quick Start. Success signal: The Sign up for your Developer Edition form is visible. Sources: S2, S8.
2. Complete the signup prompts and select Sign me up. Success signal: A signup confirmation message appears. Sources: S7, S8, S9, S10.
3. Wait for the Salesforce Developer Edition activation or welcome email to arrive in the accessible inbox. Success signal: The Salesforce activation message with its account-verification link is present. Sources: S7, S10.
4. Open the activation email and select its Verify Account or verification link. Success signal: Salesforce prompts the developer to finish registration. Sources: S7, S10.
5. Set the Developer Edition password, select and answer the security challenge question, and complete the login process. Success signal: The new Developer Edition org opens while logged in. Sources: S7, S10.
6. Confirm the first Developer Edition login succeeds and verify the user profile has API Enabled permission. Success signal: The org is accessible and API Enabled is present for the user. Sources: S2, S10.
7. Install or update the latest Salesforce CLI and verify the installation. Success signal: Salesforce CLI verification succeeds. Sources: S3.
8. Run sf org login web, sign in to the Developer Edition org in the opened browser, and select Allow. Success signal: The CLI prints Successfully authorized with the username and org ID. Sources: S3.
9. Run sf org display --target-org with the username and copy the Access Token and Instance Url. Success signal: Org Description shows Status Active plus Access Token and Instance Url. Sources: S3.
10. Prepare cURL requests with the instance My Domain, straight quotes, Authorization: Bearer token, JSON content type where needed, and local request-body files for writes. Success signal: The shell preserves the full bearer token and can read the body file. Sources: S4.
11. GET /services/data/ on the org's My Domain to list available Salesforce API versions. Success signal: The response contains label, url, and version entries. Sources: S5.
12. GET /services/data/v67.0/ with the bearer token to list resources for the documented version. Success signal: The response includes sobjects, search, query, and recent resource URLs. Sources: S5.
13. GET /services/data/v67.0/sobjects/ to run Describe Global and confirm Account is available. Success signal: The response includes the Account object's metadata and URLs. Sources: S5.
14. GET /services/data/v67.0/sobjects/Account/ to retrieve basic Account metadata and recent items. Success signal: The response includes Account name, label, key prefix, capabilities, and recent Account IDs if present. Sources: S5.
15. GET /services/data/v67.0/sobjects/Account/describe/ to retrieve detailed Account fields and relationships. Success signal: The response includes fields such as Id and updateable, childRelationships, and Account URLs. Sources: S5.
16. Execute the documented SOQL query SELECT name from Account and choose an Account ID from the response. Success signal: The response has done true, totalSize, and Account records with IDs. Sources: S5.
17. Create patchaccount.json with BillingCity set to Fremont and PATCH /sobjects/Account/{ID} using --data-binary. Success signal: The API returns HTTP/1.1 204 No Content. Sources: S4, S5.
18. Open or refresh the Account page in Salesforce and verify BillingCity is Fremont. Success signal: The Account billing address shows Fremont. Sources: S5.

## Documented friction gates

- Signup requires identity, job title, company, geography, accessible work email, a globally unique username where prompted, and submission of the live form. Stage 2. Evidence state: documented requirement. Sources: S7, S8, S10.
- The current form requires agreement to the Developer Services, Salesforce Program, and API terms. Stage 2. Evidence state: documented requirement. Sources: S8.
- The developer must wait for an activation email sent to an accessible inbox. Stage 3. Evidence state: documented requirement. Sources: S7, S10.
- The activation-email verification link must be opened before password setup and first login. Stage 4. Evidence state: documented requirement. Sources: S7, S10.
- Activation requires a new password and security challenge question; username, password, and login URL must be preserved. Stage 5. Evidence state: documented requirement. Sources: S7, S10.
- The representative route requires installing and verifying the latest Salesforce CLI plus cURL. Stage 7. Evidence state: documented requirement. Sources: S3, S4.
- Browser login and an Allow consent action are required before CLI authorization completes. Stage 8. Evidence state: documented requirement. Sources: S3.
- The bearer token and instance URL must be copied from CLI output and the token can expire. Stage 9. Evidence state: documented requirement. Sources: S3.
- cURL quoting, token exclamation marks, JSON files, headers, instance domain, and API version create several local failure points. Stage 10. Evidence state: documented requirement. Sources: S4, S5.
- A queryable and updateable Account record and its ID are required for the terminal PATCH. Stage 16. Evidence state: documented requirement. Sources: S5.
- The authenticated user needs object, record, and BillingCity update access to receive 204. Stage 17. Evidence state: documented requirement. Sources: S2, S5.
