**Developer goal:** Create a Stytch account and project, select an official authentication implementation, and test it until the guide's login or signup flow produces an authenticated session.

**Meaningful success:** The local React app renders Stytch UI and completes a Consumer Authentication login or signup, producing session and user data.

**Evidence boundary:** demonstrated-terminal-state. An authenticated session is the first proof that project configuration, frontend SDK, Stytch UI, and the selected auth method work end to end.

## Documented path

1. Open Stytch's general Quickstart. Success signal: Implementation type, framework, and authentication project-type controls are displayed. Sources: S1.
2. Select Stytch UI, React, and Consumer Authentication. Success signal: The React Stytch UI instructions are visible. Sources: S1, S6.
3. Sign up through the official Stytch dashboard. Success signal: The dashboard exposes the Test environment public token. Sources: S2, S3, S5.
4. Enable Frontend SDKs, authorize the local development domain, and copy the Test public token. Success signal: The public token and authorized domain are configured. Sources: S2, S6.
5. Install the Stytch React SDK, initialize the provider with the public token, and render the documented Stytch UI login/signup component. Success signal: The login/signup form appears. Sources: S6.
6. Run the app and complete a login or signup through the rendered Stytch UI. Success signal: The auth flow completes and returns to the app. Sources: S6.
7. Verify the React app receives and displays authenticated session or user data. Success signal: Session and user hooks return authenticated data. Sources: S6.

## Documented friction gates

- The developer must select implementation type, framework, and Consumer versus Multi-tenant Authentication before Stytch exposes a concrete guide. Stage 2. Evidence state: documented requirement. Sources: S1.
- A Stytch account is required before API keys or public tokens are available. Stage 3. Evidence state: documented requirement. Sources: S2.
- The client-rendered dashboard hides the public account, workspace, and initial-project transitions. Stage 3. Evidence state: documented requirement. Sources: S2, S3.
- Each selected guide requires environment-specific credentials and project configuration before code execution. Stage 3. Evidence state: documented requirement. Sources: S2, S4, S6.
