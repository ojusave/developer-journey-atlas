**Developer goal:** Run Plaid locally in Sandbox, create the first Item through Link, exchange its token, and retrieve its accounts.

**Meaningful success:** The developer runs the main React Quickstart against Sandbox, creates a first Item through Link, exchanges its public token for an access token and item ID, then uses the access token to retrieve account names and balances.

**Evidence boundary:** explicitly-named. It proves the complete first web integration: credentials, two local processes, Link token, institution login, Item creation, token exchange, protected server credential use, and a real product-neutral account API response. Product customization and production access come afterward.

## Documented path

1. Open Plaid's main Quickstart and confirm that the desired outcome is a general web Link Item and account API call. Success signal: Quickstart setup and Create your first Item are visible. Sources: S1, S2.
2. Sign up for or sign in to the Plaid Dashboard, verify the account email, and acknowledge Plaid's terms. Success signal: Account verification and terms acknowledgment complete, and Developers > API Keys is accessible. Sources: S1, S4, S7.
3. Open the Dashboard API Keys section and obtain the team's live Client ID plus the Sandbox secret. Success signal: The live team client_id plus environment secrets are available, and the Sandbox pair is selected for local configuration. Sources: S1, S4.
4. Verify Git, npm, the intended backend runtime, and a Unix-capable terminal are available. Success signal: git, npm, and the selected language commands run. Sources: S1, S2.
5. Clone the official plaid/quickstart repository and enter its root directory. Success signal: frontend plus node, python, ruby, go, and java directories are present. Sources: S1, S2.
6. Copy .env.example to .env and populate at least PLAID_CLIENT_ID and the Sandbox PLAID_SECRET. Success signal: The selected backend can read both required values. Sources: S1, S2, S3.
7. Choose Node, Python, Ruby, Go, or Java as the first-party Quickstart backend. Success signal: One first-party backend directory and start command are selected. Sources: S1, S2.
8. Enter quickstart/node and run npm install. Success signal: npm completes and the backend dependencies are present. Sources: S1, S2.
9. Run ./start.sh for the Node backend. Success signal: The backend reports that it is listening on localhost:8000. Sources: S1, S2.
10. In a second shell enter quickstart/frontend and run npm install. Success signal: The frontend packages are present. Sources: S1.
11. Run npm start in quickstart/frontend. Success signal: The frontend serves http://localhost:3000. Sources: S1, S2.
12. Open http://localhost:3000 and select Launch Link. Success signal: The Plaid Link institution-selection UI opens. Sources: S1, S2.
13. Let the backend call /link/token/create for the Quickstart user and return the one-time link_token. Success signal: The React client initializes Link with the returned link_token. Sources: S1, S4.
14. Select a supported Sandbox institution in Link. Success signal: Link displays the institution's test credential fields. Sources: S1, S3.
15. Enter user_good and pass_good, and enter 1234 if Link prompts for two-factor authentication. Success signal: Link advances through account connection and reaches success. Sources: S1, S3.
16. Complete Link so it creates the first Item and invokes onSuccess with public_token. Success signal: The Quickstart frontend receives public_token through Link onSuccess. Sources: S1, S5.
17. Allow the frontend to send public_token to the Quickstart backend. Success signal: The Node backend receives the public token. Sources: S1, S2.
18. Let the backend call /item/public_token/exchange and capture access_token and item_id. Success signal: The backend logs or returns public_token_exchange complete with no Plaid error. Sources: S1, S4, S5.
19. Keep the access token on the server for the Quickstart session. Success signal: The backend can use the token for the session's Plaid API calls. Sources: S1, S5.
20. Select the Quickstart Accounts button. Success signal: The Node backend calls Plaid /accounts/get with the stored access token. Sources: S1, S2.
21. Verify the /accounts/get result is rendered in the Quickstart. Success signal: The page displays account names, balances, Item metadata, and request_id. Sources: S1, S2, S4.

## Documented friction gates

- A Plaid team account, email verification, and terms acknowledgment precede Dashboard credential access; exact dynamic signup and use-case screens are not described in the Quickstart. Stage 2. Evidence state: documented requirement. Sources: S1, S4, S7.
- The team Client ID and Sandbox secret must be retrieved from the Dashboard and kept server-side. Stage 3. Evidence state: documented requirement. Sources: S1, S4.
- The Quickstart requires Git, npm, an active backend runtime, two local processes, and Unix-style shell behavior; Windows also needs symbolic-link handling. Stage 4. Evidence state: documented requirement. Sources: S1, S2.
- Environment, credentials, requested products, institution support, and optional redirect URI must agree or Link and API requests fail. Stage 6. Evidence state: documented requirement. Sources: S1, S2, S3.
- Five first-party backend languages have different installs and failure modes, with no canonical language label. Stage 7. Evidence state: documented requirement. Sources: S2.
- The end-user simulation requires institution selection, Sandbox username and password, and sometimes 2FA code 1234 before an Item exists. Stage 7. Evidence state: documented requirement. Sources: S1, S3.
- The public token must be exchanged server-side, and the resulting long-lived access token must never be exposed to the client. Stage 7. Evidence state: documented requirement. Sources: S1, S4, S5.
- Most products work immediately in the free Sandbox, but some products require the team to request special enablement before the first test Item can be created. Stage 3. Evidence state: documented requirement. Sources: S3.
