**Developer goal:** Start without a JFrog account, open the official cloud trial, provision a JFrog Platform deployment, create a Generic local repository, create an access token, upload a small file with the Artifactory REST API, and receive HTTP 201 Created.

**Meaningful success:** A developer-controlled file is stored at a chosen path in a newly created Generic local Artifactory repository.

**Evidence boundary:** explicitly-named. This is the earliest observable terminal in the selected official quickstart. The workshop policy permits treating signup as one opaque step and then reconstructing the publicly documented post-login path.

## Documented path

1. Open JFrog Start Free and choose the 14-day Cloud trial. Success signal: The page identifies no credit card as required. Sources: S1.
2. Create the JFrog Cloud trial as one opaque signup and provisioning step. Success signal: Artifactory opens in the new deployment. Sources: S1.
3. Open Administration > Repositories and create a Local repository of type Generic. Success signal: Repository Key and project settings are available. Sources: S2.
4. Enter a unique Repository Key and required project or stage settings, then create the repository. Success signal: The repository appears in Artifactory. Sources: S2.
5. From the Artifactory profile, generate and copy an identity token. Success signal: The reference token is displayed for copying. Sources: S5.
6. Create or choose a small local file and its target repository path. Success signal: The file is readable locally. Sources: S3, S4.
7. PUT the file bytes to `/artifactory/{repoKey}/{filePath}` with bearer authentication. Success signal: The API returns HTTP 201 Created. Sources: S4.
8. Confirm the deployment response status. Success signal: HTTP status is 201 Created. Sources: S4.

## Documented friction gates

- The developer must create a JFrog identity through GitHub or email before a writable cloud deployment is available. Stage 2. Evidence state: documented requirement. Sources: S1.
- The public form does not expose the complete verification and credential sequence after the initial identity submission. Stage 2. Evidence state: documented requirement. Sources: S1.
- The developer must choose a deployment hostname, cloud provider, and region. Stage 3. Evidence state: documented requirement. Sources: S1.
- Starting the cloud trial requires acceptance of the trial terms and privacy processing. Stage 3. Evidence state: documented requirement. Sources: S1.
- A writable JFrog Platform deployment must be provisioned before authenticated repository work can begin, but the public page does not quantify the wait. Stage 3. Evidence state: documented requirement. Sources: S1.
- The later local-repository form requires a unique Repository Key and repository settings, including a project stage. Stage 3. Evidence state: documented requirement. Sources: S2.
- The API route requires bearer authentication; the self-service identity-token flow displays a reference token for the developer to copy before closing the generation window. Stage 3. Evidence state: documented requirement. Sources: S4, S5.
- The authenticated user must have deploy permission for the target repository. Stage 3. Evidence state: documented requirement. Sources: S4.
