## Documented path

1. Open the Resend Introduction and follow the Get started sequence. Success signal: Create an API key and Add a domain are visible before the quickstart list. Sources: S1.
2. Create or sign in to a Resend account as a human. Success signal: API keys and Domains pages are accessible. Sources: S5.
3. Open the Dashboard API keys page and select Create API Key. Success signal: Key configuration fields are visible. Sources: S2.
4. Enter a descriptive API-key name of no more than 50 characters. Success signal: The name is accepted. Sources: S2.
5. Select Sending access unless full API access is required. Success signal: A permission is selected. Sources: S2.
6. Optionally restrict a Sending access key to a specific domain, then create it. Success signal: The re_ key value is displayed. Sources: S2.
7. Copy the one-time API key into secure storage. Success signal: The key is available for the selected quickstart. Sources: S2.
8. Enter an owned domain, preferably a purpose-specific subdomain. Success signal: The domain is accepted. Sources: S3.
9. Choose the sending region closest to most recipients. Success signal: A region is selected. Sources: S3.
10. Optionally set a custom Return-Path subdomain; otherwise keep the send.example.com default. Success signal: The domain can be added with default or custom Return-Path. Sources: S3.
11. Copy the generated DKIM and SPF TXT and MX records into the DNS provider. Success signal: The records are publicly visible, which can be checked with dns.email. Sources: S3.
12. Wait for Resend to verify the domain. Success signal: The domain becomes verified in the Dashboard. Sources: S3.
13. In the Node.js project, install the resend package with npm install resend. Success signal: resend is present in the project dependencies. Sources: S4.
14. Create the project's .env file and set RESEND_API_KEY to the one-time re_ key saved in the shared path. Success signal: process.env.RESEND_API_KEY resolves to the intended active key. Sources: S2, S4.
15. Create server.ts, import Resend, and initialize new Resend(process.env.RESEND_API_KEY). Success signal: server.ts can construct the client without a missing environment value. Sources: S4.
16. In server.ts call resend.emails.send with from Acme <onboarding@resend.dev>, to delivered@resend.dev, subject Hello World, and HTML <strong>It works!</strong>. Success signal: The message fields and success-or-error handling match the quickstart. Sources: S4.
17. Execute server.ts using the Node.js or TypeScript runner configured for the project. Success signal: The program completes the resend.emails.send call. Sources: S4.
18. Verify that the program logs the returned data object rather than error. Success signal: The Resend SDK returns and logs data for the Hello World send to delivered@resend.dev. Sources: S4.

## Documented friction gates

- A human must create the Resend account; the public docs do not expose the exact signup sequence. Stage 2. Evidence state: documented requirement. Sources: S5.
- The secret is displayed once and requires permission and optional domain-scope decisions. Stage 6. Evidence state: documented requirement. Sources: S2.
- The developer must own a domain and have permission to change its DNS. Stage 8. Evidence state: documented requirement. Sources: S3.
- Generated DKIM and SPF TXT and MX records must be copied exactly into an external DNS provider. Stage 12. Evidence state: documented requirement. Sources: S3.
- Verification often takes 15 minutes but can take 72 hours, with manual restart only after the long wait. Stage 13. Evidence state: documented requirement. Sources: S3.
- Fourteen code or CLI quickstarts plus multiple AI-tool routes are available without a universal default. Stage 14. Evidence state: documented requirement. Sources: S1, S5, S6.
