**Developer goal:** Create the minimum QuickBooks Online developer setup, obtain a temporary sandbox access token in the OAuth 2.0 Playground, call the Accounting API CompanyInfo entity, and inspect the sample-shaped response.

**Meaningful success:** The developer creates the required QuickBooks Online developer resources, uses the OAuth 2.0 Playground to obtain a temporary access token and sandbox realm ID, sends the documented GET CompanyInfo request, and receives a CompanyInfo object for the sandbox company.

**Evidence boundary:** explicitly-named. It is Intuit's explicitly documented first request, proving developer access, app and scope setup, sandbox availability, temporary OAuth authorization, bearer-token use, realm selection, and a successful Accounting API response. SDK integration, durable OAuth handling, writes, webhooks, publication, and production access come later.

## Documented path

1. Open Create and start developing your app and select the QuickBooks Online Accounting API path. Success signal: Steps 2 through 6 and the highlighted quick-sample-call section are visible. Sources: S1.
2. Create the Intuit Developer account and profile, entering the requested basic company information and a workspace name. Success signal: My Hub, App dashboard, and sandbox resources are accessible. Sources: S1.
3. Create a developer app and grant it the accounting scope. Success signal: The app opens in App dashboard and exposes Keys and credentials plus Settings. Sources: S1, S4.
4. Use the sandbox QuickBooks Online company that Intuit automatically creates with the developer profile. Success signal: The sandbox appears under My Hub, Sandboxes and can be selected by the Playground. Sources: S1, S3.
5. Confirm the app's Development Client ID and Client Secret in Keys and credentials. Success signal: Both development credential values are visible and can be copied securely. Sources: S1, S2, S3.
6. Open the OAuth 2.0 Playground, select the sandbox app or company, select the accounting scope, obtain an authorization code, and exchange it for tokens. Success signal: The Playground response contains accessToken and the UI exposes the Realm ID. Sources: S1, S4.
7. Copy the documented GET CompanyInfo cURL command and replace both sandbox-company placeholders with the Realm ID and the bearer placeholder with the access token. Success signal: No REPLACE_WITH placeholders remain and the realm ID is identical in both URL positions. Sources: S1, S3, S4.
8. Send the GET CompanyInfo request to the sandbox QuickBooks Online Accounting API. Success signal: The server returns JSON rather than an OAuth, scope, realm, or endpoint error. Sources: S1, S4.
9. Compare the response shape with Intuit's sample server response. Success signal: The response matches the sample's CompanyInfo response structure for the selected sandbox. Sources: S1.

## Documented friction gates

- QuickBooks Online Accounting, QuickBooks Payments, QuickBooks Desktop, API Explorer, generic cURL, and language-specific demo routes have different first-mile paths. Stage 1. Evidence state: documented requirement. Sources: S1, S5.
- The developer account requires basic company information and a workspace name; the current form sequence is not documented field-by-field. Stage 2. Evidence state: documented requirement. Sources: S1.
- A developer app and accounting scope must exist before the Playground can issue a CompanyInfo-capable token. Stage 3. Evidence state: documented requirement. Sources: S1, S4.
- The first request depends on a selectable sandbox company and its immutable region if a new sandbox must be created. Stage 4. Evidence state: documented requirement. Sources: S1, S3.
- Development and Production credentials are separate; Production credentials introduce a questionnaire and approval and do not belong in the sandbox path. Stage 5. Evidence state: documented requirement. Sources: S2.
- The Playground requires app and sandbox selection, accounting scope, a Connect authorization prompt when shown, code generation, and token exchange. Stage 6. Evidence state: documented requirement. Sources: S1, S4.
- The bearer token, sandbox realm ID in two URL positions, sandbox hostname, and JSON headers must all be correct in the raw cURL request. Stage 7. Evidence state: documented requirement. Sources: S1.
- Playground access tokens are temporary and valid for 60 minutes; expiry requires refresh or reauthorization before retrying. Stage 8. Evidence state: documented requirement. Sources: S4.
