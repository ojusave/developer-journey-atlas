**Developer goal:** Create a Zoom application or integration and reach the first local test state documented by Zoom.

**Meaningful success:** A development General App is authorized and added to the developer's own Zoom account for local testing.

**Evidence boundary:** demonstrated-terminal-state. This is the terminal state demonstrated by Zoom's official General App Quick start guide. Making a REST API request, implementing an SDK feature, sharing a beta app, or publishing is not part of that guide's terminal local-test step.

## Documented path

1. Open the Zoom Developer Platform home and select Start building. Success signal: The build-flow page opens. Sources: S1, S3.
2. Review the build-flow overview and its What, Where, and Which questions. Success signal: The developer can proceed to the Quick start guide. Sources: S3.
3. Read Key concepts as required by the build-flow prerequisites. Success signal: The stated knowledge prerequisite is completed. Sources: S3, S4, S6.
4. Go to the Zoom App Marketplace and sign in with a Zoom account. Success signal: The Marketplace loads for the signed-in account. Sources: S4, S10.
5. Select Developer in the lower-left Marketplace navigation. Success signal: The Developer link is visible and opens Created apps. Sources: S4, S5.
6. On Created apps, select Develop and then Build an app. Success signal: The app creation screen is shown. Sources: S4.
7. Select General app and then Create. Success signal: The Basic Info page for the new app is available. Sources: S4.
8. Generate a generic app name and development app credentials. Success signal: The generated values appear on Basic Info. Sources: S4.
9. Optionally replace the generated app name and save by clicking outside the name field. Success signal: The new name is displayed. Sources: S4.
10. Confirm whether the app is admin-managed or user-managed. Success signal: Basic Info displays the chosen management type. Sources: S4, S7.
11. Enter the required development OAuth Redirect URL or endpoint. Success signal: The redirect URL is present in OAuth Information. Sources: S4, S8.
12. Add the required OAuth allow-list URL or URLs. Success signal: The required allow-list entry is present. Sources: S4, S8.
13. Optionally enable Strict Mode URL or Subdomain check. Success signal: The selected restriction is reflected in OAuth Information. Sources: S4, S8.
14. On Features, select and configure the Zoom product and feature path required by the intended General App. Success signal: The intended feature configuration is represented in the app. Sources: S3, S4.
15. On Scopes, select Add Scopes. Success signal: Zoom product and scope choices are available. Sources: S4, S7.
16. Select the Zoom product, check the desired scopes, and select Done. Success signal: The selected scopes appear on the app. Sources: S4.
17. Explain in Scope Description why each requested scope is necessary. Success signal: The scope justification is present. Sources: S4.
18. Open Local Test in the Development view. Success signal: The Local Test wizard is visible. Sources: S3, S9.
19. Verify the app against all local-test requirements and display any incomplete fields. Success signal: No required field remains incomplete. Sources: S9.
20. Optionally select Preview Your App Listing Page. Success signal: The preview is displayed. Sources: S4, S9.
21. Select Add App Now for the developer's own account. Success signal: The Allow action is presented. Sources: S4, S9.
22. Review the authorization request and select Allow. Success signal: Zoom accepts the authorization and adds the app. Sources: S4, S9.
23. Authorize and add the app for local testing. Success signal: The app is visible in My apps for the user-managed branch, or is available for management in the web portal for the admin-managed branch. Sources: S4, S9.

## Documented friction gates

- A Zoom account is required to sign in to the App Marketplace. Stage 4. Evidence state: documented requirement. Sources: S4, S10.
- New account creation requires an email verification code after age and email entry. Stage 4. Evidence state: documented requirement. Sources: S10.
- The Developer link is exposed only to an owner, admin, or role with Zoom for developers permissions. Stage 5. Evidence state: documented requirement. Sources: S4, S5.
- The developer must choose admin-managed or user-managed, which changes products, features, scopes, permissions, and the final verification surface. Stage 10. Evidence state: documented requirement. Sources: S4, S7, S9.
- A development OAuth redirect URL or endpoint is required. Stage 11. Evidence state: documented requirement. Sources: S4, S8.
- At least one OAuth allow-list URL is required. Stage 12. Evidence state: documented requirement. Sources: S4, S8.
- The developer must map the use case to applicable Zoom products and the Access, Surface, Embed, or Connect feature path. Stage 14. Evidence state: documented requirement. Sources: S3, S4.
- Requested scopes determine allowed API methods and are subject to Zoom's stated minimal-access review principle. Stage 16. Evidence state: documented requirement. Sources: S4.
- The Local Test wizard blocks progression by listing incomplete required fields that must be resolved. Stage 19. Evidence state: documented requirement. Sources: S9.
- The developer must select Allow on Zoom's authorization screen to add the app. Stage 22. Evidence state: documented requirement. Sources: S4, S9.

## Unknowns

- Which management type is universal for a General App? Why unresolved: Zoom's official quickstart requires the developer to confirm the choice based on who will add and manage the app; it does not specify a universal default. Next evidence needed: The final verification surface differs: My apps for user-managed apps versus web-portal management for admin-managed apps.
- Which Features configuration is required for every General App? Why unresolved: The official build flow is driven by the app's What, Where, and Which answers and presents Access, Surface, Embed, and Connect as conditional paths rather than one universal configuration. Next evidence needed: The number and type of configuration steps can expand substantially before Local Test.
- Does Zoom Security Review add a wait before this local-test completion state? Why unresolved: The quickstart says that adding scopes submits a request and that unnecessary requests may be rejected, but it does not state whether or when that review blocks the documented Add App Now flow or provide a duration. Next evidence needed: A required review could lengthen the path between scope selection and app addition.
