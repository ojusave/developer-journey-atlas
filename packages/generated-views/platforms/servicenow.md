# ServiceNow developer journey

**Developer goal:** Use an authorized ServiceNow instance's REST API Explorer to create a test incident through Table API v1 and retrieve that same incident by sys_id.

**Meaningful success:** The developer opens REST API Explorer, sends a Table API v1 POST that creates an incident, captures the Location header's sys_id, sends the documented GET for that sys_id, and receives the inserted record.

**Evidence boundary:** explicitly-named. It proves instance access, the required role, Explorer navigation, API and method selection, request-body construction, a successful insert, sys_id handoff, and a successful read. Retrieve-list, update, delete, and export exercises are later.

## Documented path

1. Open the ServiceNow Developer Site, select Sign In, and on the ServiceNow ID screen select Get a ServiceNow ID. Success signal: The Sign up for a ServiceNow ID form is visible. Sources: S5.
2. Complete the ServiceNow ID signup form with First Name, Last Name, Email, Country, Password, Confirm Password, CAPTCHA, and the Terms of Use agreement, then select Sign Up. Success signal: A Thank You! message appears. Sources: S5.
3. Wait for the Welcome to ServiceNow from Developer Program! email, open it, and select Verify Email. Success signal: The verification link opens the Thank You! screen. Sources: S5.
4. On the Thank You! screen, select Login and sign in to the Developer Site with the verified ServiceNow ID. Success signal: The first-sign-in role questions appear. Sources: S5.
5. On the Help us tune your experience screen, answer Do you code? by selecting Yes (Admin), then select Next. Success signal: The job-responsibility and Developer Site terms screen appears. Sources: S5.
6. Select the option that best describes job responsibilities, add an explanation if Other is selected, open and read the Developer Site Terms of Use, close them, select the agreement checkbox, and select Finish Setup. Success signal: Developer Site setup finishes and the signed-in site header is available. Sources: S5.
7. Select Request Instance in the Developer Site header. Success signal: Available ServiceNow releases are displayed. Sources: S6.
8. Select the available release intended for the Personal Developer Instance and submit the request. Success signal: The Developer Site indicates that instance assignment is in progress. Sources: S6.
9. Wait while ServiceNow assigns and provisions the Personal Developer Instance. Success signal: My Instance or the ready dialog shows the instance URL and admin credentials. Sources: S6.
10. Retrieve the assigned instance URL and admin credentials from My Instance or the Your instance is ready! dialog, and copy the current admin password to a safe location. Success signal: The developer has the URL and admin login details. Sources: S6.
11. Select Open Instance or the instance link, sign in with the supplied admin credentials if prompted, and verify the Personal Developer Instance opens with the admin role. Success signal: The All navigation is available in the assigned instance. Sources: S2, S3, S4, S5, S6.
12. Open All, then navigate to System Web Services, REST, REST API Explorer or search for REST API Explorer, select the module, and click Explore. Success signal: Namespace, API Name, and Version filters are visible. Sources: S2.
13. In REST API Explorer, select Table API and version v1, then choose Create a record (POST). Success signal: Path Parameters and Request Body sections for Create a record are visible. Sources: S3.
14. Set the Path Parameter table to Incident (incident). Success signal: The request URL ends with /table/incident. Sources: S3.
15. In Request Body, select Add a field, choose a field and value, and optionally add more fields with the plus control. Success signal: The body contains the intended field-value pairs. Sources: S3.
16. Click Send to create the incident. Success signal: The response shows a successful status, execution time, and Location header for the created incident. Sources: S3.
17. Record the Location header and extract the 32-character sys_id at its end. Success signal: A 32-character sys_id is available for retrieval. Sources: S3, S4.
18. Select Retrieve a record (GET), keep Incident as the table, and enter the created sys_id. Success signal: The request path includes incident and the exact sys_id. Sources: S4.
19. Click Send to retrieve the inserted incident. Success signal: The response body contains a text representation of the created incident and shows status code and execution time. Sources: S4.

## Documented friction gates

- ServiceNow ID signup requires first name, last name, email, country, password, password confirmation, CAPTCHA, and Terms of Use acceptance. Stage 2. Evidence state: documented requirement. Sources: S5.
- The developer must receive the Welcome to ServiceNow from Developer Program! email and verify the email address before first login. Stage 3. Evidence state: documented requirement. Sources: S5.
- The first-sign-in Do you code? choice determines unrestricted Admin versus restricted App Engine Studio Creator PDI access. Stage 5. Evidence state: documented requirement. Sources: S5.
- Job responsibility and Developer Site Terms of Use acceptance are required to finish profile setup. Stage 6. Evidence state: documented requirement. Sources: S5.
- A currently available ServiceNow release must be selected for the PDI request. Stage 8. Evidence state: documented requirement. Sources: S6.
- The PDI request must wait for ServiceNow assignment before URL and admin credentials appear. Stage 9. Evidence state: documented requirement. Sources: S6.
- The assigned instance URL and admin credentials must be retrieved and the current password preserved. Stage 11. Evidence state: documented requirement. Sources: S6.
- rest_api_explorer, web_service_admin, or admin is required; the selected PDI Admin role satisfies this gate. Stage 11. Evidence state: documented requirement. Sources: S1, S2, S3, S4, S5.
- Namespace, Table API, version v1, method, and Incident table must all be selected in the Explorer. Stage 13. Evidence state: documented requirement. Sources: S2, S3.
- Incident field choices and values can be constrained by required fields, ACLs, and instance business rules. Stage 15. Evidence state: documented requirement. Sources: S3.
- The POST-to-GET handoff depends on preserving the Location header and extracting its 32-character sys_id. Stage 17. Evidence state: documented requirement. Sources: S3, S4.
