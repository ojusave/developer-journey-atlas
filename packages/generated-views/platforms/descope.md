## Documented path

1. Open Descope's Getting Started Quickstart Guide and select the recommended Descope Flows approach. Success signal: The guide identifies Native Implementation and Hosted Application as Flow-based paths. Sources: S1.
2. Choose Native Implementation or Hosted Application and select the application's framework route. Success signal: No single implementation and framework combination is selected for all developers. Sources: S1, S2.
3. Sign up for or sign in to Descope. Success signal: The Console project settings are accessible. Sources: S1, S3.
4. Open the project settings and copy the Project ID for the project containing the pre-created `sign-up-or-in` Flow. Success signal: The Project ID is copied and the Flow is available. Sources: S1, S2, S4.
5. Install the Descope React SDK. Success signal: The package installation completes. Sources: S2.
6. Wrap the app with `AuthProvider` and render the Descope component. Success signal: The app compiles and displays the Flow. Sources: S2.
7. Start the React app and complete a test sign-up or sign-in through the embedded Flow. Success signal: The success callback reports the user and authenticated UI is visible. Sources: S2.
8. Mark the React quickstart checkpoint as complete. Success signal: The checkpoint reads `Application is working`. Sources: S2.

## Documented friction gates

- The recommended Flow route still requires Native versus Hosted implementation and framework choices that change required actions. Stage 2. Evidence state: documented requirement. Sources: S1, S2.
- A Descope account and project are required before a Project ID and Flow can be used. Stage 3. Evidence state: documented requirement. Sources: S1, S2, S4.
- The public Console shell does not expose signup, verification, onboarding-wizard, project, or Project ID convergence states. Stage 3. Evidence state: documented requirement. Sources: S3, S4.
- The code route depends on an existing project and sign-up-or-in Flow before the SDK can show an authentication experience. Stage 3. Evidence state: documented requirement. Sources: S1, S2.
