**Developer goal:** Set up 1Password developer tools locally and complete the first secure developer workflow.

**Meaningful success:** After local app and CLI setup and creation of a 1Password Environment, the developer either reads mounted project secrets after biometric approval or completes an SSH and Git workflow with a 1Password-held key and device authentication.

**Evidence boundary:** demonstrated-terminal-state. The guide establishes a common local tool setup, then changes prerequisites, external systems, actions, and verification based on the chosen workflow. Selecting secrets over SSH and Git would be workshop-owner judgment.

## Documented path

1. Open the 1Password Developer quickstart and use its local-development path. Success signal: Steps 1 through 4 and both first-workflow choices are visible. Sources: S2.
2. Decide whether to use 1Password with a family or individually. Success signal: One official account route is selected. Sources: S5.
3. For the individual route, open 1Password.com and sign up for 1Password. Success signal: 1Password sends the account confirmation email. Sources: S5.
4. Open the email from 1Password and confirm the account. Success signal: The confirmation continues account setup. Sources: S5.
5. Save and safely prepare the Emergency Kit that 1Password asks the developer to save during account creation. Success signal: The PDF is saved somewhere the developer can retrieve independently of the current device; optionally a printed copy records the password in a safe location. Sources: S6, S7.
6. Install the 1Password desktop app for Mac, Windows, or Linux and sign in. Success signal: The app opens with the developer's account or collection in the sidebar. Sources: S2, S6, S7.
7. In the desktop app select the account or collection, open Settings > Security, and enable device-based unlock. Success signal: The chosen unlock setting is on. Sources: S2.
8. Install 1Password CLI using Homebrew, winget, or the current manual release. Success signal: The op command is available in the terminal. Sources: S2.
9. Open desktop Settings > Developer and select Integrate with 1Password CLI. Success signal: Integrate with 1Password CLI is selected. Sources: S2.
10. Run op vault list and approve the prompted sign-in. Success signal: The terminal lists vaults the account can access. Sources: S2.
11. In the desktop app open Developer > Environments and create a project-specific Environment. Success signal: The new Environment opens in the app. Sources: S2.
12. Populate the Environment by importing an existing .env file or adding variables manually. Success signal: The variables are visible inside the Environment. Sources: S2.
13. Choose Securely load developer secrets or Simplify your SSH and Git workflows as the first workflow. Success signal: One of the two official first workflows is selected. Sources: S2.
14. In the 1Password desktop app open Developer > Environments and select the project-specific Environment created in the shared path. Success signal: The saved project variables and destination controls are visible. Sources: S2.
15. Select Destinations, then select Configure destination for local .env file. Success signal: The device-path control is available. Sources: S2.
16. Choose the device directory and filename where the local .env destination should appear. Success signal: The intended directory and .env filename are shown in the destination settings. Sources: S2.
17. Select Mount .env file. Success signal: 1Password reports the destination as mounted or ready for access. Sources: S2.
18. In a terminal change to the configured device directory. Success signal: The shell is in the directory containing the chosen mounted filename. Sources: S2.
19. Run cat followed by the mounted .env filename. Success signal: The desktop app prompts to authorize the access when approval is required. Sources: S2.
20. Approve the 1Password authorization prompt with the configured device-based unlock. Success signal: The terminal command resumes and receives the mounted file contents. Sources: S2.
21. Verify the terminal output contains the expected variable names and values from the 1Password Environment. Success signal: The expected environment variables are printed with their saved values. Sources: S2.

## Documented friction gates

- Account setup first branches between family and individual use; the family route can depend on an organizer and invitation. Stage 2. Evidence state: documented requirement. Sources: S5.
- The individual route signs up on 1Password.com, then requires email confirmation and a strong account password. Current dynamic form and billing fields are not enumerated by the support guide. Stage 3. Evidence state: documented requirement. Sources: S5.
- The Emergency Kit must be saved safely because it contains the Secret Key and Setup Code needed for first sign-in on a new device; 1Password cannot recover the Secret Key. Stage 6. Evidence state: documented requirement. Sources: S6, S7.
- The intended local path requires Touch ID, Windows Hello, or Linux system authentication and repeated human approval. Stage 8. Evidence state: documented requirement. Sources: S2.
- Both the desktop app and CLI must be installed, with package-manager and PATH differences across operating systems. Stage 9. Evidence state: documented requirement. Sources: S2.
- Desktop-to-CLI integration must be enabled before biometric CLI authentication works as documented. Stage 10. Evidence state: documented requirement. Sources: S2.
- Environments are labeled beta, and the guide does not document account, plan, or rollout availability conditions. Stage 12. Evidence state: documented requirement. Sources: S2.
- A real project secret must already exist or be created before the loading workflow can demonstrate value. Stage 13. Evidence state: documented requirement. Sources: S2.
- Secret loading and SSH/Git are co-equal first workflows with different platforms, external systems, and terminal signals. Stage 14. Evidence state: documented requirement. Sources: S2.
- Mounted .env is documented only for Mac and Linux; Windows must use programmatic CLI or SDK access for Environments. Stage 14. Evidence state: documented requirement. Sources: S2.
- SSH/Git requires a key, external Git-service registration, agent and client configuration, and Windows OpenSSH compatibility where applicable. Stage 14. Evidence state: documented requirement. Sources: S2.
