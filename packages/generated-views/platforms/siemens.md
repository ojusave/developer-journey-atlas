**Developer goal:** Use Siemens developer technology, with the exact product and API not yet chosen.

**Meaningful success:** The portal gets a developer to product and API selection, but each Siemens product defines its own access model and result. A complete Building X Accounts API sandbox candidate reaches a successful `GET me/usergroups` response listing the UserGroups accessible to the machine user.

**Evidence boundary:** demonstrated-terminal-state. The Developer Portal and Xcelerator orientation explain discovery across Marketplace, Developer Portal, and Community but do not choose a Siemens product or technical terminal. The Accounts API response remains a candidate-only terminal.

## Documented path

1. Open the Siemens Xcelerator Developer Portal. Success signal: The four peer intent cards and Siemens product catalogue are visible. Sources: S1.
2. Choose Get Started, Read API documentation, Find an API, or View Developer Resources. Success signal: The selected discovery surface opens. Sources: S1, S3.
3. Open Building X Openness and its Getting Started guide from Siemens product documentation. Success signal: The Sandbox environment section and Get sandbox access action are visible. Sources: S2, S4.
4. Choose the Sandbox environment route because no Building X customer account or connected building is assumed. Success signal: The SiemensID authentication surface is reached. Sources: S4.
5. If no SiemensID exists, use the Sign-up tab in the connected application or SiemensID User Self Service Portal. Success signal: The form accepts identity details for registration. Sources: S5.
6. Provide family name, given name, working email address, and a password satisfying SiemensID policy. Success signal: SiemensID sends an OTP to the supplied email. Sources: S5.
7. Retrieve the emailed one-time password and enter it in the registration flow. Success signal: Email verification succeeds and the identity becomes active. Sources: S5.
8. Return to the Developer Dashboard and sign in with the activated SiemensID. Success signal: The Building X sandbox access request can be opened. Sources: S4, S5.
9. Request access to the Building X sandbox through the Developer Dashboard. Success signal: Sandbox credentials are granted immediately according to the guide. Sources: S4.
10. Open My Credentials and inspect the granted Building X sandbox credentials. Success signal: Both credential values are available. Sources: S4, S6.
11. Open the Building X Accounts API Quick Start. Success signal: The documented stages are visible. Sources: S6.
12. Set the machine-user credentials as shell environment variables. Success signal: `CLIENT_ID` and `CLIENT_SECRET` are non-empty in the shell. Sources: S6, S7.
13. Request an OAuth access token with the client credentials. Success signal: The response includes `access_token`, `token_type` equal to `Bearer`, and example `expires_in` equal to 86400. Sources: S6, S7.
14. Copy the response's `access_token` into a shell variable. Success signal: The quickstart says the developer has everything needed to start using the API. Sources: S6, S7.
15. Call the first Accounts API operation to list the caller's UserGroups. Success signal: The response returns the UserGroups available to the machine user. For each returned group, the guide identifies roles in `relationships.authorizedAs`, accessible partitions in `relationships.hasAccessTo`, and the owning customer in `relationships.ownedBy`; it says listed partitions can be used with other APIs. Sources: S6.

## Documented friction gates

- The portal offers four peer intent routes and many product families with no default platform-wide path. Stage 2. Evidence state: documented requirement. Sources: S1, S2.
- The Building X candidate requires SiemensID, but SiemensID itself does not authorize a Siemens service. Stage 3. Evidence state: documented requirement. Sources: S4, S5.
- SiemensID self-registration requires four identity fields and a password meeting length, repetition, and character-class rules. Stage 3. Evidence state: documented requirement. Sources: S5.
- The identity is not activated until the email OTP is received and entered. Stage 3. Evidence state: documented requirement. Sources: S5.
- A separate Building X access request is required in the authenticated Developer Dashboard. Credentials are documented as immediate, but public docs do not expose the form fields. Stage 3. Evidence state: documented requirement. Sources: S4.
- The Building X bearer token is sensitive and must stay out of source control. Stage 3. Evidence state: documented requirement. Sources: S7.
- The example bearer token is time-limited, usually 24 hours, and must be refreshed after expiration. Stage 3. Evidence state: documented requirement. Sources: S6, S7.
- A large UserGroups result may require pagination before all accessible partitions are known. Stage 3. Evidence state: documented requirement. Sources: S6.
