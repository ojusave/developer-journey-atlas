**Developer goal:** Connect an existing GitHub repository to CircleCI Cloud and obtain the first passing pipeline.

**Meaningful success:** CircleCI has created a project linked to the selected GitHub repository, committed or accepted a CircleCI configuration, triggered the first run, and displayed it as passing.

**Evidence boundary:** demonstrated-terminal-state. A passing first run is the quickstart's first completed CI outcome. Collaboration, config debugging, SSH reruns, and later pipeline iteration follow after this state.

## Documented path

1. Open the CircleCI signup page and select Sign Up. Success signal: Email and password fields are visible. Sources: S2.
2. Enter an email address, create a password, and submit the account signup. Success signal: The onboarding questions appear. Sources: S2.
3. Answer the questions about how the developer plans to use CircleCI. Success signal: The account can choose Find Your Team or create an organization. Sources: S2.
4. On the user homepage, select Get Started in the Start a new organization card. Success signal: The organization name field appears. Sources: S2.
5. Enter an organization name and select Let's Go. Success signal: The homepage shows a Create a project card. Sources: S2.
6. Select Get Started in the Create a project card, or Home > Create Project. Success signal: CircleCI asks what work the project will do and for a project name. Sources: S1, S2, S3.
7. Choose the project option matching the work, enter a descriptive project name, and select Next: Set up a pipeline. Success signal: The first-pipeline name field appears. Sources: S3.
8. Name the pipeline, for example build-and-test, and select Next: Choose a repo. Success signal: Available GitHub repositories or an Add option are displayed. Sources: S3.
9. If GitHub repositories are not yet available, select Add, choose GitHub, and select Authorize in GitHub. Success signal: GitHub offers all-repository or selected-repository access. Sources: S3.
10. Install and authorize the CircleCI GitHub App for all repositories or a selected subset that includes the intended repository. Success signal: The target repository appears in CircleCI's chooser. Sources: S3.
11. Select the GitHub repository to connect to the CircleCI project. Success signal: CircleCI prepares or displays a config file for the repository. Sources: S3.
12. Review the prepared CircleCI configuration, or the existing .circleci/config.yml if detected, then select Next: set up your triggers. Success signal: The trigger setup appears. Sources: S1, S3.
13. Keep the default GitHub App trigger that builds on every repository commit, or review triggers without adding more. Success signal: The final setup review becomes available. Sources: S3.
14. Review the project, pipeline, repository, configuration, and trigger, then select Commit config and run. Success signal: The new project opens on its pipelines page with a run in progress. Sources: S1, S3.
15. Wait for CircleCI to execute the first pipeline and inspect its state. Success signal: The pipeline is marked passing. Sources: S1.
16. Expand the passing workflow, select its job, and expand a step to inspect the successful output. Success signal: The selected step shows completed output rather than an error. Sources: S1.

## Documented friction gates

- CircleCI account creation requires email, password, onboarding questions, terms acceptance, privacy acceptance, and reCAPTCHA. Stage 1. Evidence state: documented requirement. Sources: S2.
- A new developer must create or join a CircleCI organization before project creation. Stage 4. Evidence state: documented requirement. Sources: S2.
- First-time GitHub App installation and repository authorization can require a GitHub organization administrator or repository admin. Stage 9. Evidence state: documented requirement. Sources: S3.
- The repository needs a valid CircleCI configuration and trigger; the current wizard prepares both but requires review and a commit or existing file. Stage 12. Evidence state: documented requirement. Sources: S1, S3.
- The selected setup writes configuration to the GitHub repository on a new branch before it can run. Stage 14. Evidence state: documented requirement. Sources: S1, S2, S3.
- The developer must wait for the first CI run to reach a passing or failed terminal state. Stage 15. Evidence state: documented requirement. Sources: S1.
