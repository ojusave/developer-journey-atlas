**Developer goal:** Stream sample data through a first Confluent Cloud Kafka topic and see the messages.

**Meaningful success:** A Basic Confluent Cloud Kafka cluster is running, a users topic exists, a Datagen connector is producing sample records, and those records are visible in the topic's Messages tab.

**Evidence boundary:** demonstrated-terminal-state. It is the first externally observable streaming-data result in the vendor's official Cloud Quick Start. Stream Lineage and Flink SQL follow afterward.

## Documented path

1. Create a Confluent Cloud account with local email and password, Google, or GitHub, then enter the required profile and country details and accept the applicable terms. Success signal: The developer is signed in to the Cloud Console and the organization receives its trial credit. Sources: S3, S4, S5, S6.
2. Sign in to Confluent Cloud and click Add cluster. Success signal: Basic cluster configuration is available. Sources: S1.
3. Select Begin configuration for Basic, choose cloud provider, region, and one availability zone, then continue. Success signal: The Create cluster page shows configuration and cost information. Sources: S1.
4. Name the cluster, review cost information, and select Launch cluster. Success signal: Cluster Overview displays. Sources: S1.
5. Open Topics, click Create topic, enter `users`, and select Create with defaults. Success signal: A success message confirms topic creation. Sources: S1.
6. Configure the Datagen Source connector for the users topic. Success signal: Connector status changes from Provisioning to Running. Sources: S1.
7. Open Topics, select users, and open its Messages tab. Success signal: Messages are displayed in the topic detail page. Sources: S1.

## Documented friction gates

- Confluent Cloud signup requires a local email route or Google or GitHub identity, plus profile and country details; local signup adds email verification and password creation. Stage 1. Evidence state: documented requirement. Sources: S1, S3.
- The organization automatically receives $400 in credit for up to 30 days. The trial ends when credit is exhausted or expires, and the organization is suspended if no payment or new promotion is available. Stage 1. Evidence state: documented requirement. Sources: S4, S5.
- Cluster creation displays cost information and requires setting a payment method when none exists; created resources can be billable. Stage 3. Evidence state: documented requirement. Sources: S1, S2.
- Cloud provider, region, and availability zone must be selected. Stage 3. Evidence state: documented requirement. Sources: S1.
- Cluster provisioning can take a few minutes. Stage 4. Evidence state: documented requirement. Sources: S1.
- The connector needs a generated Kafka API key and secret, which are downloaded during setup. Stage 6. Evidence state: documented requirement. Sources: S1.
