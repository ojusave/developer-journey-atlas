**Developer goal:** Create a first Box Platform application and make a first authenticated API call without a preselected authentication method or tool.

**Meaningful success:** A newly activated Box Free Developer account uses its automatically created and authorized CCG application to obtain an access token and retrieve the authenticated service-account user.

**Evidence boundary:** demonstrated-terminal-state. This is the shortest complete current official path from no account to a successful authenticated Box API response, using the app and authorization state Box creates by default.

## Documented path

1. Open the Box developer signup page, enter name and email, create a password, accept the terms, and select Get Started. Success signal: The Developer Console is available immediately and shows the default app. Sources: S6.
2. Open the default CCG app's Configuration and collect the client ID, client secret, and enterprise ID needed to authenticate as its service account. Success signal: All token-request credentials are available. Sources: S6, S7.
3. POST the CCG credentials to `https://api.box.com/oauth2/token` with `grant_type=client_credentials`, `box_subject_type=enterprise`, and the enterprise ID as `box_subject_id`. Success signal: The token endpoint returns an `access_token`. Sources: S7, S9.
4. Send `GET https://api.box.com/2.0/users/me` with `Authorization: Bearer <ACCESS_TOKEN>`. Success signal: Box returns HTTP 200. Sources: S8.
5. Inspect the response JSON and confirm it identifies the authenticated Box service-account user. Success signal: A user object with fields such as `id`, `type`, `name`, and `login` is present. Sources: S8.

## Documented friction gates

- A Free Developer account requires name, email, password, terms acceptance, and Get Started, but activates immediately without billing. Stage 1. Evidence state: documented requirement. Sources: S6.
- CCG requires client ID, protected client secret, enterprise ID, and Box 2FA to reveal the secret. Stage 2. Evidence state: documented requirement. Sources: S7.
- The token endpoint must return an active access token; server-auth tokens do not include a refresh token and must be requested again after expiry. Stage 3. Evidence state: documented requirement. Sources: S7, S9.
- Scopes, application access, enterprise authorization, endpoint restrictions, and token-user content permissions all affect success. Stage 4. Evidence state: documented requirement. Sources: S3, S8.
- The first call is complete only when Get Current User returns HTTP 200 and a user object. Stage 5. Evidence state: documented requirement. Sources: S8.
