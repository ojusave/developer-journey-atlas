**Developer goal:** Create a Zendesk Support development trial, enter Agent Workspace as an admin or agent, execute the promoted browser-console Ticketing API quickstart, and verify the three generated tickets.

**Meaningful success:** Use the JavaScript console inside a signed-in Zendesk Agent Workspace to execute the official loop of three Create Ticket API POST requests and verify the resulting Test ticket 1, Test ticket 2, and Test ticket 3 records.

**Evidence boundary:** explicitly-named. The first created ticket is the homepage's named platform milestone, while the quickstart deliberately creates three in one snippet and provides two verification surfaces. Deleting test data and exploring other APIs occur after the demonstrated success.

## Documented path

2. Open Create an account and review the free trial terms shown on the registration page. Success signal: Step 1 of 9 asks for a work email. Sources: S1, S3, S4.
3. Enter a work email in trial step 1. Success signal: The form advances to personal details. Sources: S4.
4. Enter first name and last name. Success signal: The form advances to phone number. Sources: S4.
5. Enter a phone number. Success signal: The form advances to company country. Sources: S4.
6. Select where the company is based. Success signal: The form advances to company name. Sources: S4.
7. Enter the company name. Success signal: The form advances to job title. Sources: S4.
8. Enter the developer's job title. Success signal: The form advances to employee count. Sources: S4.
9. Select the company employee-count band. Success signal: The form advances to subdomain selection. Sources: S4.
10. Choose a Zendesk subdomain. Success signal: The form accepts the subdomain and advances to password creation. Sources: S4.
11. Create a password and select Create your Zendesk account. Success signal: Zendesk accepts the registration submission. Sources: S3, S4.
12. Provision the free 14-day Zendesk Support trial and any required identity confirmation. Success signal: The developer can sign in at the chosen Zendesk subdomain as an administrator or agent. Sources: S2, S3, S4.
13. Sign in to the Zendesk Support instance as the trial admin or an agent and enter Agent Workspace. Success signal: Agent Workspace is loaded in the browser. Sources: S2.
14. As administrator, open Admin Center > Apps and integrations > APIs > API configuration for first-time API activation. Success signal: The terms checkbox and Get started action are available if API access is not already active. Sources: S5.
15. Read and accept the Zendesk terms and Application Developer and API Agreement, then select Get started. Success signal: API access settings become available. Sources: S5.
16. Evaluate the quickstart's stated API password access prerequisite against the current account. Success signal: No authoritative current page explains whether the same-origin browser-session request needs any API access toggle beyond initial terms activation. Sources: S2, S5, S6.
17. Open the browser's JavaScript console from Agent Workspace. Success signal: The console prompt is available on the Zendesk page. Sources: S2.
18. Paste the official three-iteration ticket-creation script into the JavaScript console. Success signal: The complete loop appears at the console prompt. Sources: S2.
19. Press Enter to run the script. Success signal: Zendesk accepts each request and creates a ticket. Sources: S2.
20. Verify the console lists the returned tickets as JavaScript objects. Success signal: Three ticket objects appear in the JavaScript console. Sources: S2.
21. Reload the Agent Workspace browser tab and open Views > Recently updated tickets. Success signal: Test ticket 1, Test ticket 2, and Test ticket 3 are visible in Agent Workspace. Sources: S2.

## Documented friction gates

- The documentation-home path requires a work-email-based 14-day Zendesk Support trial when no suitable instance exists. Stage 3. Evidence state: documented requirement. Sources: S1, S2, S4.
- The trial form collects personal and company data across nine steps and submits under Zendesk's Privacy Notice; marketing consent is separately presented. Stage 3. Evidence state: documented requirement. Sources: S4.
- The developer must select an available Zendesk subdomain that becomes the instance origin. Stage 10. Evidence state: documented requirement. Sources: S4.
- The account must be provisioned and usable, but the developer docs and static form do not document the email, verification, screening, or post-submit transition. Stage 12. Evidence state: documented requirement. Sources: S3, S4.
- The quickstart requires an agent or administrator role and a signed-in Agent Workspace session. Stage 13. Evidence state: documented requirement. Sources: S2.
- First-time API activation requires acceptance of Zendesk Terms of Service and the Application Developer and API Agreement. Stage 15. Evidence state: documented requirement. Sources: S5.
- The quickstart's named password-access prerequisite conflicts with current API settings for new accounts, and no current official page resolves whether the browser-session script needs a substitute toggle. Stage 16. Evidence state: documented requirement. Sources: S2, S5, S6.
- Browser or organizational policy must allow developer tools, console paste, page scripts, and same-origin API POSTs. Stage 17. Evidence state: documented requirement. Sources: S2.

## Unknowns

- What replaces the quickstart's API password access prerequisite for a new July 2026 trial? Why unresolved: The quickstart still requires and links password access, while current help says team-member password access is deprecated and unsupported for new accounts. The script supplies no explicit credentials and runs same-origin inside Agent Workspace. Next evidence needed: The stated prerequisite is unavailable to new team-member accounts, so a literal reader cannot establish whether to activate API terms only, enable API tokens, create OAuth, enable end-user password access, or do nothing beyond the signed-in browser session.
- What exact post-submit actions occur before a new trial owner reaches Agent Workspace? Why unresolved: The registration page exposes nine input stages but the developer account guide only links to the live form and does not document provisioning or the first signed-in transition. Next evidence needed: Email confirmation, anti-abuse review, launch setup, product selection, data-region choice, or guided onboarding could add pages and waits before the quickstart.
- Does the advertised 10-minute duration include any account or API-access preparation? Why unresolved: The quickstart label is 10 minutes, but its What you'll need section treats account, role, API access, and browser as pre-existing prerequisites. Next evidence needed: A new developer's actual time to first ticket can be much longer than the homepage label.
- Is creating three test tickets in a real Support instance acceptable to the developer's organization? Why unresolved: The docs recommend a trial or sponsored test account and later deletion but do not inventory instance-specific automation side effects. Next evidence needed: The quickstart mutates ticket data and can trigger business rules, notifications, routing, reporting, or compliance workflows in a configured account.
