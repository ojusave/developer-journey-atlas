**Developer goal:** Follow Marqeta's promoted Core API Quick Start from no account through a simulated sandbox transaction.

**Meaningful success:** The developer simulates the documented $10 VISA authorization in the public sandbox and can view that card transaction in Transaction Timeline.

**Evidence boundary:** explicitly-named. Core API Quick Start explicitly teaches sandbox transaction simulation after the required account, sandbox, user, card-product, and card setup; webhook and spend-control work is presented under Next steps.

## Documented path

1. Open Marqeta documentation and select Core API Quick Start. Success signal: Core API Quick Start is open. Sources: S1, S2.
2. Open the official account-creation link from Core API Quick Start. Success signal: The form displays identity, password, consent, and reCAPTCHA controls. Sources: S2, S3, S4.
3. Enter email, first name, last name, password, and password confirmation; choose whether to receive news, accept the API Terms of Service, complete reCAPTCHA, and select Create account. Success signal: The page says a verification email will arrive in a few minutes. Sources: S4.
4. Open the Marqeta account-verification email and follow its activation link. Success signal: The activation page says the email has been verified and offers Return to sign in. Sources: S4.
5. Return to sign in and authenticate with the newly created account credentials. Success signal: The developer returns to Marqeta and can access the public sandbox flow. Sources: S2, S4.
6. Open Core API Quick Start and select its account sign-up link. Success signal: The account form is visible. Sources: S1, S2, S4.
7. Enter email, first name, last name, a conforming password, and matching password confirmation. Success signal: All required identity and password fields are populated without validation errors. Sources: S4.
8. Choose the optional news setting, accept the API Terms of Service, complete reCAPTCHA, and select Create account. Success signal: Marqeta says an account-verification email will arrive in a few minutes. Sources: S4.
9. Send the new-account verification email. Success signal: The verification message arrives at the submitted address. Sources: S4.
10. Open the activation link, wait for Marqeta to validate the token, then select Return to sign in. Success signal: The page states that the email has been verified. Sources: S4.
11. Sign in again with the verified account credentials. Success signal: The public sandbox creation flow is available. Sources: S2, S4.
12. Select Create sandbox. Success signal: Marqeta begins creating the public sandbox. Sources: S2, S3.
13. Provision the public sandbox, send the readiness email, and refresh the creation page to the Dashboard. Success signal: The Dashboard displays personal API keys and a cURL to try the sandbox. Sources: S2, S5.
14. In the signed-in POST /users widget, paste the published `Marqeta User` JSON, send the request, and save the response `token`. Success signal: The Response body contains a user `token`. Sources: S2.
15. In the signed-in GET /cardproducts widget, send the request and save the `token` for the card product named Reloadable Card. Success signal: The Response body contains the named card product and its token. Sources: S2.
16. In the signed-in POST /cards widget, paste the published single-use-card JSON, replace the user and card-product token placeholders, send the request, and save the response card `token`. Success signal: The Response body contains the card `token`. Sources: S2.
17. Add the published $10 authorization JSON to the simulate-a-transaction cURL and replace the card-token placeholder. Success signal: The cURL contains the real card token and the published authorization body. Sources: S2, S5.
18. Execute the simulate-a-transaction cURL against the public sandbox. Success signal: The transaction is created for the selected card and becomes available in Transaction Timeline. Sources: S2.
19. Open Transaction timeline in the Dashboard and select the active card token. Success signal: Transaction Timeline builds and displays the simulated transaction history for the card. Sources: S2, S5.

## Documented friction gates

- Account creation requires API Terms of Service acceptance and displays a reCAPTCHA. Stage 3. Evidence state: documented requirement. Sources: S4.
- The password must have at least 16 characters with lowercase and uppercase letters, a number, and a symbol. Stage 3. Evidence state: documented requirement. Sources: S4.
- The developer must wait for and follow an account-verification email before signing in. Stage 4. Evidence state: documented requirement. Sources: S4.
- Official guidance conflicts on whether account creation already creates the sandbox and user or the developer must select Create sandbox afterward. Stage 5. Evidence state: documented requirement. Sources: S2, S3.
- The explicit Create sandbox route requires waiting for a readiness email and Dashboard refresh. Stage 5. Evidence state: documented requirement. Sources: S2.
- Asset requests can use signed-in documentation widgets or Dashboard credentials with sample cURLs. Stage 5. Evidence state: documented requirement. Sources: S2, S5.
