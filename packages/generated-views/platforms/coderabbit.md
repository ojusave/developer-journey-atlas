**Developer goal:** Receive the first CodeRabbit code-review result without a preselected review surface or Git platform.

**Meaningful success:** CodeRabbit is connected to at least one hosted repository and posts actionable review feedback on a newly opened pull request.

**Evidence boundary:** explicitly-named. The first actionable hosted review is the documented product result immediately after the common setup path. Provider-specific installation details do not change that common boundary.

## Documented path

1. Open CodeRabbit and sign up with the existing supported Git-provider account that owns or can administer the target repositories. Success signal: CodeRabbit advances to repository access selection. Sources: S1, S4.
2. Choose the organization or personal account and grant CodeRabbit access to all repositories or selected repositories. Success signal: The setup page displays `You're all set`. Sources: S1, S4.
3. Create a reviewable change in an authorized repository and open a pull request against the intended base branch. Success signal: The pull request is open in the connected provider. Sources: S1, S2.
4. Wait a few minutes while CodeRabbit analyzes the pull request and posts its review. Success signal: A CodeRabbit review appears on the pull request. Sources: S1, S2.
5. Open the CodeRabbit review and confirm it contains actionable findings or suggested improvements tied to the pull-request changes. Success signal: At least one concrete review finding, suggestion, or explicit clean review result is visible on the pull request. Sources: S1, S2.

## Documented friction gates

- A supported Git provider and owning organization or account must be selected for the hosted path. Stage 1. Evidence state: documented requirement. Sources: S1, S4.
- Hosted onboarding requires owner-level organization and repository permissions and grants CodeRabbit read and write capabilities. Stage 2. Evidence state: documented requirement. Sources: S2, S4.
- CLI route requires installing the open-beta CLI and reloading the shell. Stage 1. Evidence state: documented requirement. Sources: S3.
- CLI authenticates through browser sign-in; hosted providers use provider authorization and can require service-account tokens. Stage 1. Evidence state: documented requirement. Sources: S2, S3, S4.
- CLI must run inside an initialized Git repository and may need an explicit non-main base branch. Stage 1. Evidence state: documented requirement. Sources: S3.
- Review analysis is asynchronous; CLI agent guidance documents review durations that can exceed several minutes, though the basic Quickstart only claims setup in two minutes. Stage 4. Evidence state: documented requirement. Sources: S1, S3.
