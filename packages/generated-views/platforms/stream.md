## Documented path

1. Open Stream's developer documentation and inspect the product-level starting choices. Success signal: The developer can see that Stream does not expose a single product-neutral quickstart. Sources: S1, S2, S3.
2. Choose Chat Messaging, Video & Audio, Activity Feeds, AI Moderation, or Vision Agents, then choose that product's implementation route. Success signal: One documented branch is selected for end-to-end execution. Sources: S1, S2, S3, S6, S7.
3. For an own-application route, create or sign in to the common Stream account and choose Dashboard or Stream CLI onboarding. Success signal: The Dashboard is open or getstream init is ready to authenticate. Sources: S4, S8, S9.
4. Create, clone, pick, or temporarily provision the Stream application through the selected Dashboard or CLI path. Success signal: The application is created or selected and its credentials are available to the Dashboard or local project. Sources: S8, S9, S13.
5. Open the new application's overview and verify its common setup output before product-specific code begins. Success signal: The overview is visible or local credential file exists, and the selected product quickstart can use the app. Sources: S8, S9.
6. Confirm a React 19-compatible local toolchain and npm are available for the versions tested by the tutorial. Success signal: The Vite React TypeScript scaffold can run locally. Sources: S4.
7. Run npm create vite chat-example -- --template react-ts, then cd chat-example. Success signal: The shell is inside the generated chat-example project. Sources: S4.
8. Run npm i stream-chat stream-chat-react. Success signal: Both packages are installed in chat-example. Sources: S4.
9. Use the tutorial-generated apiKey, userId, userToken, and optional userName values shown with the code sample. Success signal: All required placeholder values can be replaced in App.tsx. Sources: S4.
10. Replace src/App.tsx with the tutorial's initial Client Setup code and substitute the generated credentials. Success signal: App.tsx has no remaining credential placeholders and matches the initial setup. Sources: S4.
11. Run npm run dev. Success signal: Vite prints a local site URL and the application loads without a terminal build error. Sources: S4.
12. Open the Vite site and verify both the ready text and a secure WebSocket connection to Stream in browser developer tools. Success signal: The page displays Chat with client is ready! and the Network panel shows the secure WebSocket connection. Sources: S4.

## Documented friction gates

- Five top-level products lead to incompatible outcomes, with no platform-wide default. Stage 2. Evidence state: documented requirement. Sources: S1, S2, S3, S6, S7.
- Each product adds SDK, UI-versus-client, client-versus-server, version, use-case, sample-app, REST, or agent-route choices. Stage 2. Evidence state: documented requirement. Sources: S1, S2, S3, S4, S5.
- Credential shape changes by route and can include API key, API secret, user JWT, token provider, and external Google AI key. Stage 2. Evidence state: documented requirement. Sources: S4, S5, S6, S7.
- Every code route requires language and package tooling appropriate to the selected SDK. Stage 2. Evidence state: documented requirement. Sources: S3, S4, S5, S6, S7.
- Video and voice routes can require camera and microphone permission plus a user gesture for browser autoplay policy. Stage 2. Evidence state: documented requirement. Sources: S5, S7.
- Own-app routes require a Stream account and app, while Vision Agents also requires Google AI Studio; the React Chat tutorial can bypass Stream signup using generated credentials. Stage 2. Evidence state: documented requirement. Sources: S4, S6, S7, S8.
- Common own-app onboarding branches between Dashboard and Stream CLI, with an additional temporary guest-account path in the CLI. Stage 3. Evidence state: documented requirement. Sources: S8, S9.
- Dashboard app creation requires a name and, for Chat and Feeds, a server region that is locked after creation; the developer also chooses development or production mode and can optionally clone an existing app. Stage 4. Evidence state: documented requirement. Sources: S8, S13.
- Chat, Video, and Moderation each offer no-card first use, but their entitlements differ and are captured separately: Chat usage limits, Video monthly API credit, and Moderation monthly content credit. Stage 2. Evidence state: documented requirement. Sources: S10, S11, S12.
