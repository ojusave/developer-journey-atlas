**Developer goal:** Create a Klarna playground account, generate test API credentials, integrate the recommended web in-line widget, and complete one test payment until Klarna returns an order ID.

**Meaningful success:** A test customer completes the Klarna widget flow and the server exchanges the authorization token for a Klarna order.

**Evidence boundary:** demonstrated-terminal-state. Klarna structures the direct web integration as initiate a payment, check out, and create an order. Order management is linked as the next section, so the first returned test order is the documented integration terminal.

## Documented path

1. Open Klarna's Before you test guide. Success signal: The guide lists Merchant portal access, test API keys, and sample data. Sources: S1.
2. Click Log in in the Klarna Docs navigation. Success signal: Region and environment controls are available. Sources: S1.
3. Select the credential region, select Playground, and click Sign up. Success signal: An email field and SIGN-UP action are available. Sources: S1.
4. Enter an email address and click SIGN-UP. Success signal: The confirmation email arrives. Sources: S1.
5. Open the confirmation email and click Activate account. Success signal: Terms, password, and name setup are presented. Sources: S1.
6. Accept the Terms & Conditions, choose a new password, enter a name, and log in to the Merchant portal. Success signal: The developer is authenticated in the test Merchant portal. Sources: S1.
7. Open Payment settings, select Klarna API Keys, and click Generate new Klarna API key. Success signal: The key dialog offers the .txt file. Sources: S1.
8. Download the .txt file and close the dialog. Success signal: Both credential values are stored securely. Sources: S1.
9. Create a one-time payment session with an authenticated POST to the regional playground `/payments/v1/sessions` endpoint. Success signal: The response contains a client_token for the JavaScript SDK. Sources: S3.
10. Add Klarna's JavaScript SDK script and an asynchronous callback to the checkout page. Success signal: The SDK can be initialized. Sources: S4.
11. Call `Klarna.Payments.init` with the session's client_token. Success signal: Initialization completes before load is called. Sources: S4.
12. Add the `klarna-payments-container` element and call `Klarna.Payments.load` with its selector. Success signal: The callback returns `show_form: true` and the Klarna option is displayed. Sources: S4.
13. Select Klarna in the checkout and trigger the Buy action using Klarna's sample customer data. Success signal: The test customer can log in and choose a Klarna payment option. Sources: S1, S4.
14. Call `Klarna.Payments.authorize` with the required customer details when the Buy action occurs. Success signal: The authorize response contains an authorization_token. Sources: S4.
15. Send the authorization token in an authenticated POST to the session's `/order` endpoint within 60 minutes. Success signal: The response returns order_id, fraud_status, and authorized_payment_method. Sources: S5.
16. Verify that the create-order success response contains `order_id`, `fraud_status`, and `authorized_payment_method`. Success signal: The documented example shows `fraud_status: ACCEPTED`, but the success contract returns the fraud-status field rather than guaranteeing that example value for every order. Sources: S1, S5.

## Documented friction gates

- The developer must choose the credential region and Playground environment. Stage 3. Evidence state: documented requirement. Sources: S1.
- A verified playground Merchant portal account is required. Stage 4. Evidence state: documented requirement. Sources: S1.
- Klarna requires Terms & Conditions acceptance during account activation. Stage 6. Evidence state: documented requirement. Sources: S1.
- The API Secret is only displayed in the generated file once. Stage 8. Evidence state: documented requirement. Sources: S1.
- Klarna pre-assesses availability and only displays the payment option when load returns show_form true. Stage 12. Evidence state: documented requirement. Sources: S4.
- The order must be created within 60 minutes of authorization. Stage 15. Evidence state: documented requirement. Sources: S5.
