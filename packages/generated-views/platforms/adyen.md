**Developer goal:** Build Adyen's recommended default online-payment flow for a website, make one test card payment through Web Drop-in, and confirm the authorized result in both the client and webhook.

**Meaningful success:** The developer creates an Adyen test account and credentials, implements Web Drop-in using the Sessions flow, submits a documented test card payment, shows the authorized result to the shopper, and receives the matching AUTHORISATION webhook with success true.

**Evidence boundary:** demonstrated-terminal-state. It is the first documented end-to-end proof that the account, server credential, client credential, session request, payment UI, shopper action, client result handling, and asynchronous webhook all work together. Live-account approval and production configuration are explicitly later steps.

## Documented path

1. Open Adyen's general getting-started guide and select Online payments as the integration family. Success signal: The online-payments checklist and build guide are the active documentation path. Sources: S1, S2.
2. Create a free Adyen test account and sign in to the test Customer Area. Success signal: The developer can open the test Customer Area and see the company and merchant account context. Sources: S1, S2.
3. Determine the test company-account and merchant-account structure, confirm the admin user, and add team users or payment methods needed for the test. Success signal: A merchant account name is available for the Sessions request and the chosen test payment method can be offered. Sources: S2, S4.
4. Generate and securely store the test API key for the payment API credential. Success signal: The key is stored outside client code and the credential has the default payment roles. Sources: S4, S5.
5. Generate the client key, add the website's allowed origin, store the key, and save the credential changes. Success signal: The test client key is available to the client application and the origin appears in the credential configuration. Sources: S4, S6.
6. Expose a webhook endpoint, configure a Standard webhook with default event codes in the test Customer Area, and test that Adyen receives a successful acknowledgement. Success signal: Test configuration succeeds and the endpoint acknowledges the test event. Sources: S2, S4, S7.
7. Choose the Sessions server-side flow, Web client platform, Drop-in UI, current Web version, and one documented server language. Success signal: The matching tabs and version in the dynamic integration guide are selected. Sources: S3, S4.
8. Install the selected official Adyen server API library and initialize its client with the API key and TEST environment. Success signal: The application can construct the Checkout service without exposing the API key to the browser. Sources: S4.
9. Create a server route that makes a test Checkout API /sessions request and returns the session id and sessionData to the client. Success signal: The Checkout test endpoint returns HTTP 201 and the client receives both session fields. Sources: S4.
10. Install Adyen Web on the front end using the recommended npm route or the documented embedded script and stylesheet route. Success signal: The browser build resolves the library and stylesheet without version or module errors. Sources: S4.
11. Add a DOM container for Drop-in, configure AdyenCheckout, create Drop-in, and mount it in the browser. Success signal: The Drop-in UI is mounted and ready for shopper interaction. Sources: S4, S6.
12. Implement redirect-result handling and shopper-facing payment-status handling. Success signal: A returned shopper sees the status corresponding to the resultCode, and non-returning flows still have webhook outcome handling. Sources: S4.
13. Open the checkout, select card, enter an official Adyen test card number, future expiry date, and test security code, then submit the payment. Success signal: Drop-in completes the payment flow and triggers a payment-completed or payment-failed event. Sources: S4, S8.
14. Verify the client-side result is Authorised and show the successful status to the shopper. Success signal: The client resultCode is Authorised, or the one-time session-result response is completed with Authorised. Sources: S4.
15. Receive the asynchronous AUTHORISATION webhook, acknowledge it, and match it to the test order. Success signal: The webhook endpoint records AUTHORISATION with success true for the same test payment and replies 200 or 202. Sources: S4, S7.

## Documented friction gates

- Four general integration families lead to materially different first miles, and Adyen does not name one platform-wide canonical route. Stage 1. Evidence state: documented requirement. Sources: S1.
- The online route requires an Adyen test account and Customer Area access; the exact current signup fields and verification sequence are not described in the documentation. Stage 2. Evidence state: documented requirement. Sources: S1.
- Company and merchant account structure, users, roles, and test payment methods must be set before the integration can create the intended checkout. Stage 3. Evidence state: documented requirement. Sources: S2, S4.
- Merchant admin and Manage API credentials roles govern access to keys and credentials. Stage 4. Evidence state: documented requirement. Sources: S4, S5.
- The API key is displayed for copying during generation and must be stored securely on the server. Stage 4. Evidence state: documented requirement. Sources: S5.
- The client key and exact allowed origin must match; newly added origins can take a few seconds to propagate. Stage 5. Evidence state: documented requirement. Sources: S6.
- End-to-end proof requires a public webhook endpoint with allowed ports, supported TLS, authentication or HMAC handling, and a 2xx acknowledgement within 10 seconds. Stage 6. Evidence state: documented requirement. Sources: S7.
- Server flow, client platform, Drop-in versus Components, Web version, and one of seven server-language routes must be chosen before implementation. Stage 7. Evidence state: documented requirement. Sources: S3, S4.
- Each server API library has its own runtime floor, package manager, install command, and client construction. Stage 8. Evidence state: documented requirement. Sources: S4.
- The Sessions request must align merchant account, currency, minor-unit amount, return URL, unique reference, environment, and payment-method context. Stage 9. Evidence state: documented requirement. Sources: S4.
- Browser support, client-side initialization, iframe and WebView guidance, allowed origin, CSP, library CSS, and redirect constraints can prevent the payment UI or shopper actions from working. Stage 11. Evidence state: documented requirement. Sources: S4, S6.
- The developer must simulate a shopper with official test credentials and may encounter redirects, 3D Secure, refused scenarios, or session expiry before authorization. Stage 13. Evidence state: documented requirement. Sources: S4, S8.
- The authoritative payment outcome arrives asynchronously by webhook; pending methods or delayed delivery extend the journey. Stage 15. Evidence state: documented requirement. Sources: S4, S7.
