**Developer goal:** Create an Airtable account, obtain or create a workspace and base with a table, create a scoped personal access token, and make the documented first list-records request.

**Meaningful success:** The intended terminal is an authenticated request to `https://api.airtable.com/v0/{baseId}/{tableName}` that returns the first page under the `records` key.

**Evidence boundary:** demonstrated-terminal-state. This is the guide's first demonstrated API response; pagination and writes follow it.

## Documented path

1. Open Airtable's Web API getting-started guide. Success signal: The request URL and response shape are documented. Sources: S1, S4.
2. Sign up for Airtable at the official signup URL. Success signal: The Airtable home screen opens. Sources: S3.
3. Create or select an accessible workspace, base, and table containing at least one record. Success signal: The base and table open in Airtable. Sources: S1, S5.
4. Open Developer Hub, choose Personal access tokens, and select Create token. Success signal: Token name, scopes, and resources can be configured. Sources: S2.
5. Name the token, add `data.records:read`, add the target base as a resource, create the token, and copy its secret. Success signal: The generated token secret is shown. Sources: S2, S4.
6. Identify the base ID and table name or table ID. Success signal: The API URL can be formed. Sources: S1, S4.
7. Send an authenticated GET request to `https://api.airtable.com/v0/{baseId}/{tableName}`. Success signal: The response contains the expected record in the `records` array. Sources: S1, S4.

## Documented friction gates

- The no-account signup application does not expose its complete public transition. Stage 2. Evidence state: documented requirement. Sources: S3.
- Workspace, base, and token capabilities depend on the user's collaborator role and organization settings. Stage 2. Evidence state: documented requirement. Sources: S1, S2, S5.
- The developer must configure `data.records:read` and the target base resource before the PAT can list records there. Stage 2. Evidence state: documented requirement. Sources: S2, S4.
- Free workspaces receive 1,000 API calls per month and all plans are subject to five requests per second per base. Stage 2. Evidence state: documented requirement. Sources: S1.
