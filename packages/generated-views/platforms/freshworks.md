**Developer goal:** Create a unified Freshworks developer account, select and create a Freshworks product trial, install the Freshworks CLI, generate the common starter app, run its local test server, and verify the app is rendered inside the selected product.

**Meaningful success:** The intended terminal is the `common-starter-template` app running through the local FDK server and rendered inside the selected Freshworks product account after `?dev=true` is appended to the account URL.

**Evidence boundary:** explicitly-named. The test guide ends the first app test at the visible product render and then moves to understanding and extending the app.

## Documented path

1. Open Freshworks Getting Started and identify the ordered signup, subscription, CLI, create-app, and test-app modules. Success signal: Sign up, product access, CLI setup, Create your first app, and Test Your First App are available as ordered modules. Sources: S1, S2, S3, S4, S5, S6.
2. Review the Sign up module's two registration methods and product-account requirement. Success signal: Identity methods, email verification, tour choice, product selection, and account naming are described. Sources: S2.
3. Sign up for or sign in to the Freshworks Developer Platform. Success signal: The developer can open `Explore Products`. Sources: S2, S7.
4. Choose a Freshworks product, name its account, and start the included trial. Success signal: The product account can be opened and its URL is known. Sources: S2, S3.
5. Install supported Node.js and Freshworks CLI, then verify the CLI. Success signal: `fdk version` reports the installed CLI. Sources: S4, S5, S6.
6. Create the starter app. Success signal: The starter template files are created. Sources: S5.
7. Run `fdk run` to start the local test server. Success signal: The server prints `http://localhost:10001/system_settings` and product test URLs. Sources: S6.
8. Configure local system settings. Success signal: The system settings or installation page completes. Sources: S6.
9. Open the product account with `?dev=true` and verify the app. Success signal: The starter app is visibly rendered in the product and its function can be tested. Sources: S6.

## Documented friction gates

- Google and email are peer registration methods with different external and verification steps. Stage 2. Evidence state: documented requirement. Sources: S2.
- Email signup requires activation through a verification email. Stage 2. Evidence state: documented requirement. Sources: S2.
- The developer must select a Freshworks product and name its account before building an app for that host. Stage 2. Evidence state: documented requirement. Sources: S2, S3.
- The live signup and authenticated product onboarding are client-rendered and do not expose the full public transition. Stage 3. Evidence state: documented requirement. Sources: S7.
- A sponsored developer subscription requires a request through Developer Assist and may introduce an external approval or wait not quantified by the guide. Stage 3. Evidence state: documented requirement. Sources: S3.
- The selected route requires compatible Node.js, Freshworks CLI, environment configuration, and the latest Chrome. Stage 3. Evidence state: documented requirement. Sources: S4, S5, S6.
- Local testing requires module selection, organization domain, valid product account URLs, and conditional installation parameters or Freshservice product type. Stage 3. Evidence state: documented requirement. Sources: S6.
