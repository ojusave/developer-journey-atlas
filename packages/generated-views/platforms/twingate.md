**Developer goal:** Protect and access a private network Resource with Twingate.

**Meaningful success:** An authenticated Twingate Client directly reaches the configured private Resource from the developer's device.

**Evidence boundary:** explicitly-named. The Quick Start places direct Resource access immediately before Congratulations and Next Steps. The workshop policy permits treating signup as one opaque official-URL step.

## Documented path

1. Open the Private Access Quick Start. Success signal: Summary of Steps is reached. Sources: S1.
2. Sign up for a free Twingate account using the official signup URL. Success signal: The Network page can be opened. Sources: S1, S2.
3. Create a Remote Network if signup did not already create one. Success signal: The Remote Network is shown in Network. Sources: S1.
4. Add the private Resource and authorize the Everyone group. Success signal: The Resource is assigned to a Group. Sources: S1, S3.
5. Select Deploy Connector and follow the displayed method for the target host. Success signal: Connection Status shows successful connections to both Controller and Relay. Sources: S1.
6. Install the Twingate Client from the official client site and authenticate. Success signal: The Client reports connected and authenticated. Sources: S1.
7. Access the configured Resource directly from the Client device. Success signal: The Resource responds directly from the device regardless of its current network. Sources: S1, S3.

## Documented friction gates

- The Quick Start requires a Twingate account, but the public signup transition is not documented. Stage 2. Evidence state: documented requirement. Sources: S1, S2.
- A Connector can be deployed only with sufficient access to a reachable Remote Network host. Stage 2. Evidence state: documented requirement. Sources: S1.
- Remote Network location, Resource details and Group, Connector deployment method, and Client platform change later required actions. Stage 2. Evidence state: documented requirement. Sources: S1.
