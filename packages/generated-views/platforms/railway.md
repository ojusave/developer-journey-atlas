## Documented friction gates

- The app path requires one Railway account per person; public docs do not reproduce the current browser signup sequence. Stage 2. Evidence state: documented requirement. Sources: S4, S5, S10.
- A user must accept Railway's Terms and meet the Terms' age requirement. Stage 2. Evidence state: documented requirement. Sources: S8.
- GitHub-based automated verification determines Full versus Limited Trial and Railway does not manually review verification requests. Stage 3. Evidence state: documented requirement. Sources: S4.
- Railway requires a linked valid GitHub account; the OAuth and repository-access sequence is external and not atomically documented in the Quick Start. Stage 7. Evidence state: documented requirement. Sources: S2.
- Deploy Now and Add Variables are peer actions. The tutorial chooses Deploy Now only for brevity. Stage 9. Evidence state: documented requirement. Sources: S2.
- The initial deploy moves through queue, build, and deploy states. Free-plan deploys are rejected during each region's peak-hours window, and Free or sometimes Hobby work may separately queue during limited-access periods. Stage 10. Evidence state: documented requirement. Sources: S3, S9.
- A trial has $5 for up to 30 days, limits projects to five services, and limits RAM and vCPU. Exhausted credit can block or stop workloads. Stage 10. Evidence state: documented requirement. Sources: S4, S7.
- For failure, inspect the build or deploy logs and read the full log. The deployment reference also provides Restart and Redeploy actions for relevant later states. Stage 12. Evidence state: documented requirement. Sources: S2, S3.
