**Developer goal:** Create a monday.com trial or developer account, enable Developer mode, copy a personal V2 API token, and make the official first GraphQL API request.

**Meaningful success:** The intended terminal is a token-authenticated POST to `https://api.monday.com/v2` containing one of the guide's board queries.

**Evidence boundary:** demonstrated-terminal-state. This is the earliest observable terminal in the selected official quickstart. The workshop policy permits treating signup as one opaque step and then reconstructing the publicly documented post-login path.

## Documented path

1. Open Making your first request. Success signal: The guide identifies trial signup as the first step. Sources: S1.
2. Create the trial account as one opaque signup step. Success signal: The monday.com workspace opens. Sources: S1, S4.
3. Optionally enable Developer mode from profile picture > monday.labs > Developer mode > Activate. Success signal: The page refreshes with Developer mode active. Sources: S1.
4. Open Developers > API token, select Show, and copy the personal V2 token. Success signal: The token is copied. Sources: S2, S3.
5. POST the guide’s `boards(limit:1){id name}` GraphQL query to `https://api.monday.com/v2` with Authorization, Content-Type, and API-Version headers. Success signal: A JSON GraphQL response is returned. Sources: S1, S2.
6. Inspect the GraphQL data response. Success signal: `data.boards` contains a board ID and name and the response includes `account_id`. Sources: S1.

## Documented friction gates

- The first-request guide presents trial and developer accounts without selecting one for a new developer. Stage 2. Evidence state: documented requirement. Sources: S1.
- Both signup destinations hide the current account and onboarding sequence in a client-rendered application. Stage 2. Evidence state: documented requirement. Sources: S4, S5.
- Only eligible confirmed users can access the API, and personal-token availability depends on user type. Stage 2. Evidence state: documented requirement. Sources: S2, S3.
- The guide includes Developer mode because it exposes IDs used by many API methods, but does not state that activation is required to obtain a personal token or send its limit-one board query. Stage 2. Evidence state: documented requirement. Sources: S1.
- Requests require the personal token in the Authorization header. Stage 2. Evidence state: documented requirement. Sources: S1, S2.
