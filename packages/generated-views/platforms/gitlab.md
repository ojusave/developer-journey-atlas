## Documented friction gates

- GitLab offers many getting-started routes without one platform-wide first success. Stage 1. Evidence state: documented requirement. Sources: S1.
- The tutorial requires the ability to sign in to a GitLab instance. Stage 2. Evidence state: documented requirement. Sources: S2.
- Self-managed administrators can disable self-registration or require administrator-driven creation or approval. Stage 2. Evidence state: documented requirement. Sources: S5.
- The first-commit route requires local Git. Stage 3. Evidence state: documented requirement. Sources: S2.
- The SSH route requires a supported OpenSSH client. Stage 4. Evidence state: documented requirement. Sources: S3.
- The developer must create and protect an SSH key pair and add the public key to GitLab. Stage 6. Evidence state: documented requirement. Sources: S3.
- The developer may need to verify the GitLab host fingerprint on the first SSH connection. Stage 12. Evidence state: documented requirement. Sources: S3.
- Project creation and default-branch write or merge permissions depend on the namespace and project policies. Stage 13. Evidence state: documented requirement. Sources: S2, S4.
- git commit requires configured author identity even though the tutorial does not separately enumerate that setup. Stage 27. Evidence state: documented requirement. Sources: S2.
- Protected-branch or merge-request policies can reject the tutorial's direct default-branch push. Stage 31. Evidence state: documented requirement. Sources: S2.
