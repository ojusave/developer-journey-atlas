## Documented path

1. Open the Wise Getting started guide and determine whether the developer is a Wise Platform partner or a Wise Business account using a personal token. Success signal: No shared API result exists at this choice point. Sources: S1.
2. Confirm that this is an existing Wise.com business account automating its own SMB payout workflow, not a partner integration acting on customer accounts. Success signal: The developer can use the business account's own API-token settings. Sources: S1, S3.
3. Log in to the Wise.com business account. Success signal: The business account opens. Sources: S3.
4. Go to Your Account, select Connect and manage apps, and select API tokens. Success signal: The Add new token control is visible. Sources: S3.
5. Select Add new token. Success signal: Wise either requests the conditional 2-step verification or proceeds to display the new token. Sources: S3.
6. Complete 2-step verification when Wise prompts for it. Success signal: Wise displays the newly created personal API token. Sources: S3.
7. Copy the personal API token immediately and store it securely before leaving the page. Success signal: The complete token is stored in a secure location available to the curl command. Sources: S3.
8. Run `curl -X GET https://api.wise.com/v1/profiles -H 'Authorization: Bearer <PERSONAL_API_TOKEN>'`. Success signal: Wise returns an HTTP response for the authenticated request. Sources: S3.
9. Verify that the request returns HTTP 200 and a profile-list response containing the profiles that belong to the authenticated Wise user. Success signal: HTTP 200 and a profile-list response are visible in the terminal. Sources: S3, S4, S5.

## Documented friction gates

- Partner access is mediated through Wise sales and implementation onboarding rather than self-service signup. Stage 1. Evidence state: documented requirement. Sources: S1.
- Partner client credentials are environment-specific and scoped to the agreed access model; personal tokens use a different limited route. Stage 1. Evidence state: documented requirement. Sources: S1.
- Wise must grant access before the partner can log into Developer Hub. Stage 1. Evidence state: documented requirement. Sources: S1.
- Enterprise, Correspondent, or Embedded model and a concrete workflow must be selected before API steps are determinate. Stage 2. Evidence state: documented requirement. Sources: S1, S2.
- Some partner integrations require mTLS, JOSE, SCA, or 2FA in addition to OAuth 2.0. Stage 2. Evidence state: documented requirement. Sources: S1.
- Rate limiting returns HTTP 429 with a `retry-after` header and requires waiting before retry. Stage 2. Evidence state: documented requirement. Sources: S1.
- The personal-token candidate requires 2-step verification before Wise displays the new token. Stage 1. Evidence state: documented requirement. Sources: S3.
- A personal API token is shown only once and must be copied and stored securely before leaving the token flow. Stage 1. Evidence state: documented requirement. Sources: S3.
