**Developer goal:** Complete GitHub's first project workflow entirely in the browser: create a repository, branch, commit, pull request, merge, and verify the published README change.

**Meaningful success:** The developer creates hello-world with a README, creates readme-edits, commits a README change, opens a pull request, merges it into main, deletes the branch, and sees the published change on the Code tab.

**Evidence boundary:** explicitly-named. It proves account access and the complete GitHub collaboration loop from repository creation through durable merged change. Profile customization, command-line Git, external collaboration, APIs, and automation follow later.

## Documented path

1. Create or sign in to a personal GitHub account and complete required email verification. Success signal: The GitHub dashboard and New repository action are accessible. Sources: S1, S2.
2. Select New repository, set Repository name to hello-world, type a short description, select Public or Private, select Add a README file, and select Create repository. Success signal: The repository Code tab shows README.md on main. Sources: S1.
3. Open the branch dropdown from main, type readme-edits, and select Create branch: readme-edits from main. Success signal: The repository branch selector shows readme-edits. Sources: S1.
4. On readme-edits, open README.md, select Edit, and write a short description about yourself. Success signal: The editor diff shows the intended addition on readme-edits. Sources: S1.
5. Select Commit changes, enter a descriptive commit message, and select Commit changes again. Success signal: The branch contains a commit and differs from main. Sources: S1.
6. Open Pull requests, select New pull request, compare readme-edits against main, and review the diff. Success signal: The compare page shows the README additions and no unintended changes. Sources: S1.
7. Select Create pull request, add a title and brief description, and select Create pull request again. Success signal: The pull request Conversation page is open with the expected diff. Sources: S1.
8. At the bottom of the pull request, select Merge pull request and then Confirm merge. Success signal: GitHub reports the pull request was successfully merged and closed. Sources: S1.
9. Select Delete branch for readme-edits. Success signal: readme-edits is removed without removing the merged commit. Sources: S1.
10. Return to the hello-world Code tab and verify the README change is published on main. Success signal: The about-yourself text appears in README.md and main is selected. Sources: S1.

## Documented friction gates

- GitHub account signup and email verification are required if the learner has no account. Stage 1. Evidence state: documented requirement. Sources: S1, S2.
- Repository owner, name, required short description, Public or Private visibility choice, and README initialization determine whether the remaining tutorial matches. Stage 2. Evidence state: documented requirement. Sources: S1.
- The branch must be created from main and selected before editing. Stage 3. Evidence state: documented requirement. Sources: S1.
- A committed difference between readme-edits and main is required to open a meaningful pull request. Stage 6. Evidence state: documented requirement. Sources: S1.
- Organization branch protection can add reviews, checks, permissions, merge queue, or conflict-resolution gates even though the personal tutorial expects none. Stage 8. Evidence state: documented requirement. Sources: S1.
