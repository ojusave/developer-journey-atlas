**Developer goal:** Start without a Rollbar account or project, choose the SDK matching an application, instrument it with the correct project access token, send a documented test occurrence, and verify that the occurrence appears in Rollbar.

**Meaningful success:** The selected Rollbar SDK sends a test error or message from the instrumented application and the resulting item becomes visible in the Rollbar dashboard.

**Evidence boundary:** demonstrated-terminal-state. The dashboard-visible occurrence is the first proof that the browser SDK, project token, transport, and Rollbar ingestion path work end to end.

## Documented path

1. Open the Browser JavaScript Quick Start. Success signal: The Quick Start and Test your installation sections load. Sources: S3.
2. Sign up for a Rollbar account using the official signup URL. Success signal: The developer can access Rollbar and create a project. Sources: S1, S2.
3. Create or select a project and open Settings > Project Access Tokens. Success signal: The token can be copied into the browser snippet. Sources: S1, S3, S4.
4. Paste the Quick Start snippet high in the page head before other scripts and replace `POST_CLIENT_ITEM_ACCESS_TOKEN`. Success signal: The instrumented page loads in a browser. Sources: S3.
5. Run the documented `window.onerror` test in the browser console. Success signal: The command runs and Rollbar receives the occurrence. Sources: S3.
6. Open the project dashboard and wait a few seconds. Success signal: The test occurrence appears in the project dashboard. Sources: S3.

## Documented friction gates

- A Rollbar account and project are required before obtaining a project token and seeing captured occurrences. Stage 2. Evidence state: documented requirement. Sources: S1, S2, S3.
- The public signup page does not expose the exact post-submission account and project states. Stage 2. Evidence state: documented requirement. Sources: S2.
- The developer must choose an SDK and often a framework-specific integration. Stage 2. Evidence state: documented requirement. Sources: S1.
- The integration requires a project token with an appropriate client or server scope. Stage 2. Evidence state: documented requirement. Sources: S3, S4.
- Newly created access tokens are shown only at creation and cannot be retrieved from the UI or API afterward, so they must be copied and stored securely. Stage 2. Evidence state: documented requirement. Sources: S4.
- The SDK must be embedded or installed and configured in application code. Stage 2. Evidence state: documented requirement. Sources: S1, S3.
- The Browser JavaScript quick start says the simulated error should appear in the dashboard in a few seconds. Stage 2. Evidence state: documented requirement. Sources: S3.
