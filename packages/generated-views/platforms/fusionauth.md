**Developer goal:** Add working login to a local Node.js application with FusionAuth.

**Meaningful success:** The registered administrator logs into the sample Express application through FusionAuth and returns to the home page with a personalized greeting.

**Evidence boundary:** explicitly-named. The 5 Minute Intro says all three installation routes end with a functioning application whose login and logout are provided by FusionAuth, and the Docker guide explicitly identifies the personalized post-login page as successful authentication. Logout and later customization follow this state.

## Documented path

1. Open Get Started and follow the 5 Minute Intro. Success signal: The Docker route is available. Sources: S1, S3.
2. Select Docker because Docker and Docker Compose are installed. Success signal: Docker Install is the active route. Sources: S2, S3.
3. Download `docker-compose.yml` and `.env` with the two published `curl` commands. Success signal: The files can be edited. Sources: S2.
4. Edit `.env` to change `DATABASE_PASSWORD` and ensure that `POSTGRES_USER` and `POSTGRES_PASSWORD` are correct. Success signal: The edited file is saved. Sources: S2.
5. Run `docker compose up -d` and then `docker compose logs -f`. Success signal: The logs show the FusionAuth HTTP server starting on ports 9011 and 9012. Sources: S2.
6. Open `http://localhost:9011` to start the Setup Wizard. Success signal: Administrator and license fields are available. Sources: S2.
7. Create the administrator, accept the license, record the chosen email and password, and submit the Setup Wizard. Success signal: The administrative UI opens. Sources: S2.
8. If the First Time Setup panel appears, click Skip. Success signal: Applications can be opened. Sources: S2.
9. Open Applications, click Add, and give the new Application a display name. Success signal: The OAuth tab can be selected. Sources: S2.
10. On the OAuth tab, add `http://localhost:3000/oauth-redirect` as the authorized redirect URL and select its popup value. Success signal: The selected value appears in the authorized redirect list. Sources: S2.
11. Set the Logout URL to `http://localhost:3000/logout`, enable Authorization Code, and set Require registration to complete OAuth authorization to true. Success signal: All four published settings are present. Sources: S2.
12. Save the Application, open its View action, and copy its Client Id and Client Secret. Success signal: Client Id and Client Secret are available for the sample app. Sources: S2.
13. Open Users, find the Setup Wizard administrator, and click Manage. Success signal: Add Registration is available. Sources: S2.
14. Click Add Registration, select the new Application if prompted, and save without additional registration data. Success signal: The guide states that the user now has permission to log into the Application. Sources: S2.
15. If the administrator has no first name, edit the user and add one for the sample greeting. Success signal: The profile shows the first name. Sources: S2.
16. Clone `fusionauth-example-5-minute-guide` and enter its directory. Success signal: Its `.env.sample` and package files are available. Sources: S2.
17. Copy `.env.sample` to `.env`. Success signal: `.env` can be edited. Sources: S2.
18. Replace `CLIENT_ID` and `CLIENT_SECRET` in `.env` with the copied values and keep `BASE_URL=http://localhost:9011`. Success signal: No `CHANGEME` values remain for the two credentials. Sources: S2.
19. Run `npm install`. Success signal: The application is ready to start. Sources: S2.
20. Run `npm start`. Success signal: `http://localhost:3000/` can be opened. Sources: S2.
21. Open `http://localhost:3000/` in an incognito browser and click Login. Success signal: The login form is shown. Sources: S2.
22. Log in with the administrator credentials created in the Setup Wizard. Success signal: The application home page reappears with the user authenticated. Sources: S2.
23. Verify that the home page greets the authenticated user by first name. Success signal: The guide states that authentication succeeded and user data was stored in the Express session. Sources: S2.

- The self-hosted Setup Wizard requires license acceptance before it creates and logs in the administrator. Stage 7. Evidence state: documented requirement. Sources: S2.
