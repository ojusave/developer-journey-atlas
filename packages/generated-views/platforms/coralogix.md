**Developer goal:** Create a first Coralogix organization and team, send one log through the direct REST API, and find that log in Explore.

**Meaningful success:** A log posted to the region-correct `/logs/v1/singles` endpoint appears in Explore.

**Evidence boundary:** explicitly-named. The matching Explore row is the first observable proof that Coralogix accepted and indexed the submitted event.

## Documented path

1. Open Coralogix documentation and select Create an account. Success signal: Account creation is listed as the first getting-started action. Sources: S1, S2.
2. Sign up with a work email at the official Coralogix signup URL. Success signal: The verification message arrives. Sources: S1, S2.
3. Open the verification message and click ACTIVATE. Success signal: The account setup page opens. Sources: S2.
4. Complete user setup, enter the organization name, and click SAVE AND CONTINUE. Success signal: The first-team prompt appears. Sources: S2.
5. Enter a team name and click CREATE NEW TEAM. Success signal: The team's Coralogix page and unique URL are available. Sources: S2.
6. Open Settings, Users and Teams, API Keys; create a Send-Your-Data key with the SendData preset and copy it. Success signal: The key appears in the API-key list and its value is retained. Sources: S3, S4, S5.
7. Determine the Coralogix domain value from the team login URL. Success signal: The endpoint can be formed as `https://ingress.\[\[DOMAIN_VALUE\]\]/logs/v1/singles`. Sources: S3, S5.
8. POST the documented JSON array to `/logs/v1/singles` with Content-Type and Bearer key headers. Success signal: The HTTP request succeeds. Sources: S5, S7.
9. Open Explore, select logs, choose a time range covering the request, filter by the submitted application name, and click Run. Success signal: A result row appears. Sources: S6, S7.
10. Inspect the matching result row. Success signal: Application, subsystem, and text match the POST body. Sources: S5, S6, S7.

## Documented friction gates

- Account creation requires work-email activation. Stage 2. Evidence state: documented requirement. Sources: S2.
- An existing organization can require an Org Admin to grant access or create a team. Stage 2. Evidence state: documented requirement. Sources: S2.
- The new-organization procedure does not document the login, redirect, team URL, or authenticated state after CREATE NEW TEAM. Stage 2. Evidence state: documented requirement. Sources: S2.
- Telemetry type and ingestion method change every downstream setup action. Stage 2. Evidence state: documented requirement. Sources: S1, S3.
- Creating and viewing Send-Your-Data keys requires SendData permissions. Stage 2. Evidence state: documented requirement. Sources: S4.
- Advanced API-key security can make the generated key visible only once and require a local download. Stage 2. Evidence state: documented requirement. Sources: S4.
- The ingestion endpoint must match the region inferred from the team login hostname. Stage 2. Evidence state: documented requirement. Sources: S3, S5.
