**Developer goal:** Start without Sonatype software or an account, download Nexus Repository Community Edition for a local test, start it, complete the initial administrator setup, create a Raw hosted repository, upload a small file, and reach the uploaded component's search results.

**Meaningful success:** A file is stored as a component in a newly created Raw hosted repository, and Nexus Repository redirects the user to search results for the uploaded component path.

**Evidence boundary:** demonstrated-terminal-state. A stored and searchable component is the first observable repository-management outcome; the policy permits selecting the documented local platform route rather than requiring one cross-platform sequence.

## Documented path

1. Open the Nexus Repository download page and select Community Edition plus the current archive for the host operating system and architecture. Success signal: A current platform-specific archive is present locally. Sources: S1.
2. Extract the downloaded archive into a suitable installation location. Success signal: The application directory contains the `nexus` start script and the data-directory parent exists. Sources: S2.
3. Start Nexus Repository with `./nexus start` or use `./nexus run` for foreground testing. Success signal: The UI becomes reachable at `http://localhost:8081/` for a default local test. Sources: S2.
4. Open the UI, sign in as `admin` with `$data-dir/admin.password`, change the password, move through the onboarding wizard, accept the Community Edition EULA, and choose anonymous access. Success signal: The onboarding wizard completes. Sources: S2, S6.
5. Open Settings > Repository > Repositories, select Create repository, choose `raw (hosted)`, provide a unique name and required storage settings, and create it. Success signal: The repository appears in the repository list. Sources: S3, S5.
6. Open Upload, choose the Raw hosted repository, select a local file, provide any required path, and click Upload. Success signal: Nexus redirects to search results for the component path. Sources: S4, S5, S6.
7. Verify the uploaded component appears in the redirected search results. Success signal: The search result contains the uploaded component path. Sources: S4.

## Documented friction gates

- Download and installation depend on operating system and processor architecture. Stage 1. Evidence state: documented requirement. Sources: S1.
- Self-hosting requires local runtime resources, writeable directories, archive extraction, and server process management. Stage 2. Evidence state: documented requirement. Sources: S2.
- The running service must be reachable at its configured host and port, and its process user needs access to the application and data directories. Stage 3. Evidence state: documented requirement. Sources: S2.
- The generated administrator password must be retrieved from the data directory and changed before normal use. Stage 4. Evidence state: documented requirement. Sources: S2.
- The first-start wizard requires an anonymous-access decision. Stage 4. Evidence state: documented requirement. Sources: S2.
- Repository creation and component upload require separate create, read, upload, edit, browse, and read privileges. Stage 4. Evidence state: documented requirement. Sources: S3, S4.
