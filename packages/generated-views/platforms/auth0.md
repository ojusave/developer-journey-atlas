**Developer goal:** Create an Auth0 environment and reach the first authenticated application or protected-API success.

**Meaningful success:** After account and tenant creation, the developer chooses an application architecture and framework, completes its Auth0 setup, and verifies that route's first authentication or authorization behavior.

**Evidence boundary:** demonstrated-terminal-state. Account and tenant creation are common, but application type changes whether the developer registers a public or confidential client, uses Universal Login, stores a client secret, configures mobile callbacks, or validates bearer tokens. A workshop owner must select the intended first success.

## Documented path

1. Open Auth0's Create Tenants getting-started guide. Success signal: Set up an Auth0 account and Create a tenant and domain are visible in order. Sources: S2.
2. Create the free Auth0 account without a credit card, using credentials or a supported social provider. Success signal: Auth0 prompts the new account to create a tenant. Sources: S2, S6.
3. Enter a unique tenant name that will become part of the Auth0 domain. Success signal: The name satisfies Auth0's rules and is available. Sources: S2.
4. Select the tenant's public-cloud region and create the tenant. Success signal: The new tenant is accessible in the Auth0 Dashboard at its assigned auth0.com domain. Sources: S2.
5. Verify Node.js 20 LTS or newer and npm 10 or newer are installed. Success signal: node and npm report versions at or above the quickstart requirements. Sources: S5.
6. Create the auth0-vanillajs directory, enter it, and run npm init -y. Success signal: package.json exists in auth0-vanillajs. Sources: S5.
7. Install Vite and configure package.json with module type and the documented dev, build, and preview scripts. Success signal: npm run dev resolves to Vite. Sources: S5.
8. Create the quickstart's index.html, app.js, and style.css files. Success signal: The three files exist with the guide's initial markup, module, and styles. Sources: S5.
9. Install @auth0/auth0-spa-js. Success signal: The Auth0 SPA SDK can be imported from app.js. Sources: S5.
10. In the Auth0 Dashboard go to Applications > Applications, select Create Application, enter a name, choose Single Page Web Application, and create it. Success signal: The application's Settings page exposes Domain and Client ID. Sources: S5.
11. Copy the Auth0 Domain and Client ID from the SPA application's Settings. Success signal: The tenant Domain and application Client ID are available locally. Sources: S3, S5.
12. Create .env.local and set the quickstart's Vite Auth0 Domain and Client ID variables to the copied values. Success signal: Vite can read both environment values. Sources: S5.
13. In the Auth0 application Settings set Allowed Callback URLs, Allowed Logout URLs, and Allowed Web Origins to http://localhost:5173, then save the changes. Success signal: All three localhost values are saved on the correct application. Sources: S5.
14. Implement the quickstart's loading, error, logged-out, logged-in, profile, Log In, redirect-callback, and Log Out states in the app files. Success signal: The app has controls and state handling for the full hosted-login round trip. Sources: S4, S5.
15. Run npm run dev. Success signal: Vite serves http://localhost:5173 without build errors. Sources: S5.
16. Open http://localhost:5173 and select Log In. Success signal: The tenant's Universal Login page opens. Sources: S4, S5.
17. Sign up or authenticate through Universal Login. Success signal: Auth0 returns the browser to the allowed localhost callback with code and state for the application to process. Sources: S4, S5.
18. Verify the SPA completes callback handling and displays the authenticated state. Success signal: The browser returns to the app, code and state are removed from the URL, the console has no Auth0 errors, and the page shows Successfully authenticated plus the user profile. Sources: S5.

## Documented friction gates

- An Auth0 account is required before tenant creation; exact verification and anti-abuse screens are not in the public guide. Stage 2. Evidence state: documented requirement. Sources: S2.
- Signup requires no credit card. The 22-day trial automatically becomes the $0 Free Plan, currently published for up to 25,000 monthly active users. Card verification applies only if a Free-plan custom domain is configured. Stage 2. Evidence state: documented requirement. Sources: S6.
- The globally unique tenant name becomes part of the authentication domain and cannot be changed or reused after deletion. Stage 3. Evidence state: documented requirement. Sources: S2.
- Region controls data locality, while the numeric tenant environment is assigned rather than selected. Stage 4. Evidence state: documented requirement. Sources: S2.
- Four application classes and 48 named quickstarts prevent a platform-wide canonical implementation. Stage 5. Evidence state: documented requirement. Sources: S1.
- Interactive routes require exact callback, logout, and origin allow-list values; mismatches cause documented redirect or session failures. Stage 5. Evidence state: documented requirement. Sources: S5.
- Public clients use Domain and Client ID, while confidential web apps require a protected Client Secret and API routes add issuer and audience configuration. Stage 5. Evidence state: documented requirement. Sources: S3, S5.
- The selected framework requires its runtime, package manager, Auth0 SDK, and local application execution. Stage 5. Evidence state: documented requirement. Sources: S1, S5.
